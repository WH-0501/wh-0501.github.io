cartographer_ros 配置

tracking_frame: SLAM算法跟随坐标系. 若使用了 IMU, 则应该是 IMU 的位置(尽管可能转动). 常用的选择是: “imu_link”

published_frame: 发布位姿的子坐标系. 例如: 若系统中有其它模块发布了 odom 坐标系, 则为 odom. 此时, odom 在 “map_frame” 中的位姿将被发布, 否则设置为 “base_link” .

odom_frame: 仅在 provide_odom_frame 为 true 时有效. 使能时, carto 发布的tf 树为: publishd_frame -> odom_frame -> map_frame.

use_odometry: 若使能, 将订阅 “odom” 主题的 odometry 消息.

use_nav_sat: 若使能, 将订阅 “fix” 主题的 NavSatFix (GPS)数据.

use_landmarks: 若使能, 将订阅 “landmarks” 主题消息.

num_laser_scans: 要订阅的 laser scan 主题 “scan” 的数目. 若有多个时, 可为 “scan_1”、”scan_2”

num_multi_echo_laser_scans: 要订阅的 multi-echo laser scan 主题 “echoes” 的数目. 若有多个时, 可为 “echoes_1”、”echoes_2”

Laser scan 与 multi-echo laser scan:
标准的 laser scan 消息只包括每个激光脉冲的单个深度和强度值(intensity). 通常是强度最强的.
multi-echo: 多回波传感器能够为激光器发出的每个脉冲返回几个回波. 例如, 拾取到从玻璃以及其后面墙壁的回波, 或者在不同距离的两个物体的边界处接收到两个用于表示每个物体距离的回波.

num_subdivisions_per_laser_scan: 将每个接收到的（多回波）激光扫描分割成的点云数. 分割 scan 能保证激光设备移动过程中 scans 不会弯曲变形. 有相应的轨迹生成器选项可以将分离的scans累积到点云中, 用来进行scan matching. 用于 SensorBridge 类中，将激光点云数据按该参数分段。

num_point_clouds: 订阅的点云主题 “points2” 的个数. 若有多个时, 可为 “points2_1”、”points2_2”

lookup_transform_timeout_sec: 使用 tf2 查找transforms时的超时时间.

Fixed ratio:

rangefinder_sampling_ratio: 测距仪消息的固定采样率.
odometry_sampling_ratio: 里程计消息的固定采样率.
fixed_frame_sampling_ratio: 固定坐标系消息的固定采样率.
imu_sampling_ratio: IMU消息的固定采样率.
landmarks_sampling_ratio: landmarks消息的固定采样率
基于 FixedRatioSampler 实现按固定频率均匀采样.

voxel_filter_size：
距离机器人较近的地方通常扫描到的点更多，远处的物体的扫描点较少。为了降低计算量，需对点云进行降采样，但简单的随机采样会导致低密度区的点变少，而高密度区的点仍然较多。因而，cartographer中采用了 voxel filter(使用体素化网格实现降采样)。 voxel_filter_size 越小，过滤后的点云更密集，计算量越大；voxel_filter_size越大，数据量越小，计算速度越快，但可能会导致数据丢失。

use_online_correlative_scan_matching： 是否首先使用 correlative scan matcher 来解决在线 scan match 问题，从而为Ceres生成一个良好的起点。（即:使用RealTimeCorrelativeScanMatcher对数据源进行预处理）。开启之后的对应配置参数参考： RealTimeCorrelativeScanMatcherOptions、 CeresScanMatcherOptions、 MotionFilterOptions。

imu_gravity_time_constant：IMU重力时间常数。 基于通过IMU观察到的重力方向的移动平均值的时间常数（以秒为单位）。 应该选择使得误差：
1.来自加速度测量误差不影响重力方向（当常数减小时变得更糟）
2.来自角速度的积分（当常数增加时变得更糟）误差得到平衡。

Real-Time Loop Closure in 2D LIDAR SLAM
该论文主要描述的是 cartographer 2D 建图。

为了实现实时的闭环，使用 branch-and-bound(分支定界法)算法来计 scan-to-submap 的匹配作为约束。

scan-to-scan 匹配算法经常用于在基于激光的SLAM方法中计算相对位姿变化。 但 scan-to-scan 的匹配会迅速累积误差。

scan-to-map 的匹配有助于限制误差的累积。(本文使用的方法)

Pixel-accurate 扫描匹配算法，进一步减少本地误差的累积。该算法虽然计算量大，但对回环检测也很有用。 用于回环检测的其它算法包括：基于直方图(histogram-based)的匹配、扫描数据中的特征检测、以及使用机器学习。

有两种常用算法可以解决局部误差累积问题：粒子滤波和graph-based SLAM。(本文使用后者)。
粒子滤波必须维持每个粒子的完整系统状态的表示(即：每个粒子都需要维护整个系统的状态,如对每个粒子保存一个完整的系统地图等)。对于 grid-based SLAM，随着地图越来越大，会变得非常消耗资源。
Graph-based 方法工作于表示位姿和特征的节点的集合之上。图的边(edges)是由观察产生的约束。有多重算法算法可以用来最小化由所有约束引入的误差。
对于大地图，基于粒子滤波的 gmapping 算法比 cartographer 算法消耗的内存会更多。

cartographer 提供了一种室内实时建图方法，该方法基于激光雷达。 激光扫描数据(scan)会插入到最佳位姿估计的 submap 中，这个最佳的位置假设在短时间内是很准确的。 scan match 发生在最近的 submap 上，因此只和最近的扫描数据有关。

综合考虑硬件资源和性能，该SLAM算法不使用粒子滤波器，而使用 pose optimization 来处理误差累积。 当一个 submap 完成后，就不会再在该 submap 中插入新的 scan 数据，该 submap 会用作 loop closure 的 scan matching 的一部分。 所有构建完成的 submap 和 scan 数据都会被用来进行 loop closure。 如果它们在当前的位姿估计上足够接近，则扫描匹配器尝试在 submap 中查找 scan。 如果在当前估计的位姿周围的搜索窗口中找到足够好的匹配，则将其作为 loop closing 约束添加到优化问题中。 每隔几秒就进行闭环检测，一个位姿点被重复访问就算达到闭环了。 这导致了软实时约束(soft real-time constraint)，即：要求闭环检测必须在新的 scan 被加入到 submap 中之前完成，否则可能会导致失败。 通过使用 branch-and-bound 算法以及对每个完成的 submap 的几个预计算的网格 来实现这一点。

BBS 算法，即：branch-and-bound scan matching 算法 (分支定界法扫描匹配) – FastCorrelativeScanMatcher3D

SPA 算法，即：Sparse Pose Adjustment 算法 (稀疏位姿调整)。PoseGraph2D 及 PoseGraph3D。

PoseExtrapolator 类：将 pose 保持一定时间以估计线速度和角速度。使用速度来推断运动。 如果有IMU/Odometry，则使用IMU和/或odometry数据来改进该推断(extrapolation)。

(旧版本cartographer使用UKF来进行数据融合来估计位姿)

如果没有 imu，则在第一帧 scan 数据处理时初始化 extrapolator，若有 imu，则在每帧 imu 数据处理时都调用 InitializeExtrapolator，但因为该接口中对 extrapolator_ 进行判断，因此不会每次都重新初始化 extrapolator。 若使用 scan + odometery 建图，也只在第一帧 scan 处理时初始化 extrapolator，在处理 odom 数据时，只是将该数据添加到 extrapolator 中。

timed_pose_queue_: 缓存经过匹配校准后的位姿点。该队列不会一直缓存，会删除间隔超过 pose_queue_duration_ 的点
cached_extrapolated_pose_: 缓存根据 odom/imu/历史位姿点(timed_pose_queue_) 推断出的位姿。该队列不会清除

odometry_imu_tracker_、extrapolation_imu_tracker_: 都指向 imu_tracker_

ImuTracker:使用来自IMU的角速度和线加速度来跟踪方向。 因为平均线性加速度（假设慢速运动）是重力的直接测量，所以滚动/俯仰虽然会导致yaw角偏离，但不会出现漂移。

AddPose：
AddPose 只在累积 num_accumulated_range_data 帧 scan 数据，进行匹配得到校准后的位姿后才会执行。将校准后的位姿存入 timed_pose_queue_ 缓存中：

  • 若 imu_tracker_ 为空, 则创建并初始化
  • 将匹配校准或的位姿点存入 timed_pose_queue_ 缓存中
  • 删除 timed_pose_queue_ 中距当前时间超过 kExtrapolationEstimationTimeSec(0.001s) 的点(前提是当前缓存中点数>2,确保删除后还有至少两个点)
  • 根据 timed_pose_queue_ 中的 pose 点及点之间的时间间隔计算线速度和角速度
  • 更新 imu_tracker_ - AdvanceImuTracker
  • TrimImuData() - 删除 imu_data_ 中的无用数据(过期数据)
  • TrimOdometryData() - 删除 odometry_data_ 中的无用数据
  • 构建 odometry_imu_tracker_ 及 extrapolation_imu_tracker_ 指针(每次执行 AddPose 都会重新 make_unique 构建)

ExtrapolatePose：推算位姿

  • 调用 ExtrapolateTranslation 方法计算偏移量
    • 若有里程计数据，则根据从里程计数据得到的线速度 linear_velocity_from_odometry_ 及时间间隔 来计算偏移量
    • 若没有里程计数据，则根据从缓存的两点计算得到的线速度 linear_velocity_from_poses_ 及时间间隔 来计算偏移量

计算得到的偏移量 + 上一点的位置信息，得到预估的点的坐标信息

  • 调用 ExtrapolateRotation 方法计算旋转量(即:偏转角度)
    • 调用 AdvanceImuTracker
    • 根据 imu_tracker 获取上次的方向 imu_tracker_->orientation()
    • 计算偏转角 last_orientation.inverse() * imu_tracker->orientation()

  • 将预估的位姿点存入 cached_extrapolated_pose_ 中

  • 若IMU为空，或IMU时间戳异常，若有 odom 数据，则使用从 odom 得到的角速度 angular_velocity_from_odometry_ 来更新位姿，若没有 odom ，则根据缓存的历史位姿点计算得到的角速度 angular_velocity_from_poses_ 来更新方向。

Closing Loops

通过 Sparse Pose Adjustment(SPA) 算法优化所有 scan 和 submap 的位姿，来实现全局优化。

LocalTrajectorBuilder 中，构建了一个个 submap，也记录了每帧 scan 插入到 submap 中的 pose。 由于传感器测量值存在误差，虽然在局部范围内(即 submap 中)误差影响不大，但由于建图是一个增量式的过程，第 n 个 pose 的预估值依赖于前面 n-1 个 pose 的估计值，因而，会导致误差的累积。因此需要 loop closure 来消除这种误差累积。

一旦局部 scan-to-submap 成功，就将该 scan 的 pose 作为节点添加到优化问题中。
