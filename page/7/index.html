CUDA

发表于 2024-09-20 更新于 2024-09-22 分类于 NVIDIA ， CUDA 本文字数： 5.8k 阅读时长 ≈ 10 分钟

CUDA 与 CUDA Driver 并非一一对应，一台电脑上只有一个版本的 CUDA Driver，但可以有多个 CUDA 版本。

CUDA API 包括 CUDA Runtime API 和 CUDA Driver API。

  • Runtime API 与 Driver 最大区别是懒加载

    • 第一个 runtime API 调用时，进行 cudaInit 初始化

    • 第一个需要 context 的 API 调用时，会调用 cudaDevicePrimaryCtxRetain 创建 context

  • cuda 开头的都是 Runtime API

  • Runtime API 对应 cuda_runtime.h 和 libcudart.so

CUDA 支持多种语言，无论哪种语言或接口，指令最终都会被 driver 转换为 PTX(Parallel Thread Execution) 代码。

CUDA 架构组件包括：

  • NVIDIA GPU 并行计算引擎
  • 对硬件初始化、配置的OS内核级支持
  • 硬件初始化、配置的OS内核级支持
  • 用于并行计算kernel和函数的PTX 指令集架构(ISA，Instruction set architecture)

CUDA Toolkit 与 conda cudatoolkit

CUDA Toolkit 是 NVIDIA 官方提供的安装包，包含了 NVIDIA Driver、CUDA 编译器、开发工具等，参考：CUDA Toolkit major components。

CUDA Toolkit 包含了 CUDA 开发调试相关的所有组件。但对 Pytorch 等深度学习框架，其大多只需要使用 CUDA 的动态链接库。

Anaconda 在安装 pytorch 等需要用到 CUDA 的框架时，会自动安装 cudatoolkit，其主要包含应用程序在使用 CUDA 时所依赖的动态库（不包括 runtime 层和以下的层，包括 nvcc 和 CUDA driver）。安装好之后，只需要 cudatoolkit 与 NVIDIA driver 兼容即可。Anaconda 安装的 cudatoolkit 位于 ~/anaconda3/pkgs/ 目录中。

pytorch 检测 CUDA 版本的代码位于 torch/utils/cpp_extension.py 中的 find_cuda_home

实际调用哪个 CUDA 库，需根据 tf/pytorch 生成时设置的链接库路径确定。如 pytorch setup.py 中指定了链接库的搜索路径，其中 CUDA_HOME 获取逻辑为：

  • 默认 cuda 安装路径 /usr/local/cuda
  • 若默认路径不存在，则搜索 nvcc 所在目录
  • 若 nvcc 也不存在，则直接寻找 cudart 库文件目录（此时通常使用 conda 安装的 cudatoolkit），将其上级目录作为 CUDA_HOME
  • 若未找到 CUDA_HOME ，则生成 CPU 版 pytorch

额外的硬件信息

计算能力没有描述某些硬件特性。例如，无论计算能力如何，大多数 GPU(但不是所有 gpu) 都可以将内核执行与主机和设备之间的异步数据传输重叠。在这种情况下，调用 cudaGetDeviceProperties() 来确定设备是否具有特定功能。例如，asyncEngineCount 字段指示是否可能重叠内核执行和数据传输(如果是，可能有多少并发传输)，canMapHostMemory 字段表示是否可以执行零拷贝数据传输。

要针对特定版本的 NVIDIA 硬件和 CUDA 软件，请使用 nvcc 的 -arch， -code 和 -gencode选项。例如，使用 warp shuffle 操作的代码必须使用 -arch=sm_30(或更高的计算能力)进行编译。

CUDA Runtime

提供以下功能

  • Device management
  • Context management
  • Memory management
  • Code module management
  • Execution control
  • Texture reference management
  • Interoperability with OpenGL and Direct3D

与底层 CUDA Driver API 相比，CUDA Runtime 通过提供隐式初始化、上下文管理和设备代码模块管理，极大地简化了设备管理。nvcc 生成的 c++ 主机代码利用 CUDA Runtime，因此链接到此代码的应用程序将依赖于 CUDA Runtime；类似地，任何使用 cuBLAS, cuFFT 和其他 CUDA toolkit 库的代码也将依赖于 CUDA Runtime。

CUDA Runtime 在内核启动之前处理内核加载和设置内核参数和启动配置。隐式驱动版本检查、代码初始化、CUDA 上下文管理、CUDA 模块管理(从库到函数映射)、内核配置和参数传递都是由CUDA Runtime 执行的。

它包括两个主要部分:

  • C 风格的函数接口(cuda_runtime_api.h)
  • C++ 风格的封装(cuda_runtime.h)

CUDA Compatibility

CUDA 软件环境由三部分组成:

  • CUDA Toolkit（库、CUDA Runtime 及开发工具）：用于开发者构建 CUDA 程序
  • CUDA Driver：用户态驱动，如 linux 下的libcuda.so
  • NVIDIA GPU device driver：内核态驱动组件

（来源：https://docs.nvidia.com/cuda/cuda-c-best-practices-guide/index.html#cuda-compatibility-developer-s-guide）

在 Linux 系统上，CUDA Driver 和内核态组件在 NVIDIA display driver 中一起提供。

为了方便起见，每个 CUDA Toolkit 附带了一个 NVIDIA display driver。该驱动程序支持该版本 CUDA Toolkit 中引入的所有功能。

通常，升级 CUDA Toolkit 包括升级 Toolkit 和 Driver，以获得最先进的工具包和驱动程序功能。

但这并不总是必需的。CUDA 兼容性允许只升级某些组件，并不总是需要一个完整的系统升级。

CUDA Toolkit 版本

从CUDA 11开始，Toolkit 版本基于行业标准的版本控制方案: .x.y.z:

  • x 代表主版本 - API 已经改变，二进制兼容性被打破
  • y 代表次要版本 - 引入新 API，弃用旧 API，源代码兼容性可能会被破坏，但保持二进制兼容性
  • z 代表发布/补丁版本

Toolkit 中各组件版本信息，可查看 CUDA Toolkit Major Component Versions

CUDA 支持几种兼容性选择：

  • CUDA 向前兼容升级最初是在 CUDA 10 中引入的，旨在让用户能够访问新的 CUDA 功能，并在安装了较旧的 NVIDIA datacenter driver 的系统上运行使用新 CUDA 版本构建的应用程序
  • CUDA 增强兼容性在 CUDA 11.1 中首次引入，提供了两个好处：
    • 通过利用 CUDA Toolkit 中组件语义版本控制，可以为一个 CUDA 小版本(例如11.1)构建应用程序，并在主要家族(即11.x)中的所有未来小版本中工作
    • CUDA Runtime 放宽了最低 driver 版本检查，因此在移动到新的次要版本时不再需要升级 driver
  • CUDA Driver 确保编译后的 CUDA 应用程序保持向后二进制兼容性。使用 CUDA Toolkit 3.2 编译的应用程序可以运行在较新的 Driver 上

源码兼容性

源代码兼容性定义为库提供的一组保证，其中针对特定版本的库(使用SDK)构建的格式良好的应用程序将在安装新版本的SDK时继续构建和运行而不会出现错误。

二进制兼容性

二进制兼容性定义为库提供的一组保证，其中针对所述库的应用程序在与库的不同版本动态链接时将继续工作。

CUDA Driver API 有一个 C-style ABI，它保证在旧驱动程序(例如CUDA 3.2)上运行的应用程序仍然可以在现代驱动程序(例如CUDA 11.0附带的驱动程序)上正常运行。

因此，CUDA Driver API 是二进制兼容的，但不兼容源代码(根据较新的 SDK 重新构建应用程序可能需要更改源代码)。

Minor Version Compatibility

CUDA 11 开始支持 Minor Version Compatibility

CUDA 11及后续版本默认为 Minor Version 兼容性。

从CUDA 11开始，使用 CUDA major 发行系列中的 CUDA Toolkit 编译的应用程序可以在具有至少最低要求的 driver 版本的系统上运行，功能集有限。这个最低要求的 driver 可以不同于 CUDA Toolkit 中打包的驱动程序，但应该属于同一个 major 版本。

由于二进制向后兼容，基于任何旧的 CUDA Toolkit 构建的应用程序总是能在新 driver 上运行，在 CUDA 11 之前，基于新 CUDA Toolkit 构建的应用程序在没有向前兼容包的旧 driver 上不受支持。

如果使用的是新的 CUDA 10。则所需的最小驱动程序版本与打包为该工具包版本一部分的驱动程序版本相同。因此，在CUDA 11.1之前，每个新的 CUDA Toolkit 小版本都更改了所需的最低 driver 版本。因此，系统管理员总是必须升级 driver ，以支持基于CUDA Toolkit 构建的应用程序。

Minor Compatibility 部署注意

如上，直接依赖于 CUDA Runtime 的应用程序可以部署在以下两种场景中:

  • 系统上安装的 CUDA driver 比 Runtime 更新
  • CUDA Runtime 比系统上的 CUDA driver 更新，但它们来自相同的 CUDA Toolkit 的 major 版本

在场景 2 中，系统管理员应该意识到上述限制，并且应该能够在遇到任何问题时指出应用程序失败的原因。

Minor 版本兼容性为库的使用和部署提供了灵活性。使用支持次要版本兼容性的库的应用程序可以部署在具有不同版本的 Toolkit 和库的系统上，而无需针对库版本的差异重新编译应用程序。这对库的旧版本和新版本都适用，前提是它们都来自同一个 major 发行系列。请注意，应该考虑库本身具有相互依赖性。例如，每个 cuDNN 版本都需要一个特定版本的 cuBLAS。

Jetson 的可升级包

参考：https://developer.nvidia.com/blog/simplifying-cuda-upgrades-for-nvidia-jetson-users/

Jetson 两种系统：

  • NVIDIA 打包好的 Jetpack
  • 基于 NVIDIA 提供的 BSP，第三方厂商自行构建的系统

这两类系统中，都自带固定版本的 CUDA 开发包、运行时库、驱动程序。

NVIDIA 从 CUDA 11.8 版本开始，才为 Jetson 提供 CUDA 升级包功能。

以前不提供，是因为基本系统里已经包含经过检验的稳定的 CUDA 驱动和系统，若用户随意调整，将对整个 Jetpack 系统及 BSP 二次开发的稳定性造成影响，厂商需要重新测试，重新校验稳定性。

但同时又存在需与频繁更新的 CUDA 开发包新功能同步的需求。因此，NVIDIA 从 Jetpack 5.0 和 CUDA 11.8 开始，对基本系统进行调整：从 CUDA 11.8 开始，引入升级路径，为 Jetson 提供了升级 CUDA 驱动和 CUDA Toolkit 的选项。

蓝色方框描述了 NVIDIA JetPack 5.0 SDK 中默认存在的组件。虚线将 Jetson Linux BSP 与 NVIDIA JetPack SDK 的其他组件分开。绿色框表示可以通过此功能升级到的 CUDA 组件。

通过引入 CUDA 驱动程序（包括 libcuda.so 及其依赖项）升级，无需在像以前一样只能使用 Jetson Linux BSP 中打包的默认 CUDA 驱动提供的功能。

升级操作参考：https://developer.nvidia.com/zh-cn/blog/simplifying-cuda-upgrades-for-nvidia-jetson-users/

引入了中间兼容层 （cuCompart)。以前追求稳定的内置驱动，是直接和 NVMAP/NVHOST/EGL/NVGPU 等更底层的组件直接交互的，所以不能轻易动，现在从 JP 5.0 起，驱动变成和 cuCompat 交互了，而 cuCompat 再和底层的 NVMAP/NVHOST/EGL/NVGPU 交互，这样就多了一个中间层次了。可以简单理解成，多了一个抽象层，从原本的驱动中抽离，这个抽象层本身可以得到测试，可以稳定。而新版本的驱动变成了它上面的一个更高一层的软件了。

TRAC-IK Kinematics Solver

发表于 2024-09-17 更新于 2024-09-22 分类于 机械臂 ， MoveIt ， MotionPlanning ， 运动学求解器 本文字数： 992 阅读时长 ≈ 2 分钟

TRAC-IK 是由 TRACLabs 开发的一款逆运动学求解器，它通过线程将两种 IK 实现结合在一起，以实现比常用的开源 IK 求解器更可靠的解决方案。从官方文档：(TRACK-ik)为 KDL 中流行的雅可比矩阵逆方法提供了一种可选的逆运动学求解器。具体地说，KDL 的收敛算法是基于牛顿方法的，这种方法在存在关节限制的情况下不能很好地工作——这在许多机器人平台中很常见。trac-IK 同时运行两种 IK 实现：

  • 一个是对 KDL 基于牛顿的 convergence 的简单扩展，该算法通过随机跳跃检测并缓解由于关节限制而导致的局部最小值
  • 第二种是 SQP(Sequential Quadratic Programming,即:顺序二次规划)非线性优化方法。使用能够更好地处理关节限制的准牛顿(quasi-Newton)方法

默认情况下，当这些算法中的任何一个收敛到一个解时，IK 搜索会立即返回。为了获得 “最佳” IK 解，还提供了距离和可操作性的次要约束条件。

trac_ik_kinematics_plugin 包提供了一个 KinematicsBase MoveIt 接口，可以替换默认的 KDL 求解器。目前不支持模拟关节。

IKFast Kinematics Solver

发表于 2024-09-17 更新于 2024-09-22 分类于 机械臂 ， MoveIt ， MotionPlanning ， 运动学求解器 本文字数： 7.5k 阅读时长 ≈ 14 分钟

机器人运动学编译器 IKFast 是 Rosen Diankov 的 OpenRAVE 运动规划软件中提供的一个强大的逆运动学求解器。与大多数逆运动学求解器不同，IKFast 可以解析任何复杂运动学链的运动学方程，并生成特定语言的文件(如c++)供以后使用。最终的结果是非常稳定的解决方案。

MoveIt IKFast 是使用 OpenRAVE 生成的 cpp 文件为 MoveIt 生成 IKFast 运动学插件的工具。本教程将引导设置机器人以利用 IKFast 的强大功能。MoveIt IKFast 插件生成器工具不适用于 >7 DOF 的手臂。

选择 IK 类型

需要选择想要的 IK。最常见的 IK 类型是 transform6d。

IK types：

  • Transform6D：end effector reaches desired 6D transformation
  • Rotation3D：end effector reaches desired 3D rotation
  • Translation3D：end effector origin reaches desired 3D translation
  • Direction3D：末端执行器坐标系的方向达到预期方向 direction
  • Ray4D：ray on end effector coordinate system reaches desired global ray
  • Lookat3D：末端执行器坐标系统的方向指向期望的三维位置
  • TranslationDirection5D：end effector origin and direction reaches desired 3D translation and direction. Can be thought of as Ray IK where the origin of the ray must coincide.
  • TranslationXY2D：end effector origin reaches desired XY translation position, Z is ignored. The coordinate system with relative to the base link.
  • TranslationLocalGlobal6D：local point on end effector origin reaches desired 3D global point. Because both local point and global point can be specified, there are 6 values
  • TranslationXAxisAngle4D,TranslationYAxisAngle4D, TranslationZAxisAngle4D： end effector origin reaches desired 3D translation, manipulator direction makes a specific angle with x/y/z-axis (defined in the manipulator base link’s coordinate system)
  • ranslationXAxisAngleZNorm4D，TranslationYAxisAngleXNorm4D，TranslationZAxisAngleYNorm4D：end effector origin reaches desired 3D translation, manipulator direction needs to be orthogonal to z, x, or y axis and be rotated at a certain angle starting from the x, y, or z axis (defined in the manipulator base link’s coordinate system)

kinematics配置

发表于 2024-09-17 更新于 2024-09-22 分类于 机械臂 ， MoveIt ， MotionPlanning ， 运动学求解器 本文字数： 2.3k 阅读时长 ≈ 4 分钟

MoveIt! 使用插件结构，尤其是允许用户编写自己的逆运动学算法。Forward kinematics（正向运动学） and finding jacobians（查找雅可比矩阵） 被整合到自己的 RobotState 类。默认逆运动学插件配置使用 KDL numerical jacobian-based solver。由 MoveIt! Setup Assistant 自动配置。

用户可以选择执行自己的运动学求解器，例如 PR2 有自己的运动学求解器。要实现这样的求解的一种流行的方法是使用 ikfast 包产生需要与特定工作的机器人的 C++ 代码。

在规划场景中，冲突检测通过 CollisionWorld 对象来配置，由 FCL 包（主要的CC库）来执行。

Allowed Collision Matrix (ACM)（免检冲突矩阵），在运动规划里，冲突检测会耗费甚至达到 90% 的计算资源。ACM 编码需要检测的对象间的对应关系（机器人的或世界的）。如果在 ACM 关联两对象的值为1，那就不需要检测，这情况就比如两个对象相隔很远，永远不会发生碰撞。

通用参数

  • kinematics_solver：运动学求解器插件名。必须与 plugin description 文件中的名字一致
  • kinematics_solver_search_resolution：指定求解器可能用于在冗余空间中搜索逆运动学的分辨率
  • kinematics_solver_timeout：逆运动学求解器可能执行的每个内部迭代的默认超时(以秒为单位)。典型的迭代(例如，数值求解器)将包括从种子状态随机重新启动，然后是求解周期(适用于此超时)。解算器可以尝试多次重启。默认的重启次数由 kinematics_solver_attempts 参数定义
  • kinematics_solver_attempts：重试次数。最好将超时设置得较低，并在单个解决方案周期中快速失败

Cached IK 插件

Cched IK 运动学插件创建 IK 解决方案的持久缓存。这个缓存然后被用来加速任何其他 IK 求解器。对 IK 求解器的调用将在缓存中使用类似的状态作为 IK 求解器的种子(seed)。如果不能返回一个解，则使用用户指定的 seed 状态再次调用 IK 求解器。若新的 IK 解决方案与缓存中的状态有很大不同，将被添加到缓存中。定期将缓存保存到磁盘。

OMPL

发表于 2024-09-17 更新于 2024-09-22 分类于 机械臂 ， MoveIt ， OMPL 本文字数： 2.1k 阅读时长 ≈ 4 分钟

OMPL：Open Motion Planning Library，是一个包含许多常用的基于采样的运动规划算法（以基于采样的算法，如RRT、PRM为主）的 C++ 库。也是 Moveit!(工具集) 默认采用的运动规划算法库。

OMPL.app 是 OMPL 的前端，它是一个包含 FCL 和 PQP 碰撞检测器和基于 PyQt/PySide 的简单 GUI 用户界面的轻量级包。

特性：基于采样，概率完备，非最优/渐进最优。

核心概念

运动规划问题核心概念

  • 状态空间：即 构型空间，内部的点表征着规划系统的特点状态。对于一个自有运动的刚体，其状态空间由三个平移状态和三个旋转状态构成；而对于具有 n 个转动关节的机械臂，状态空间可被表征为一个 n 维的圆环面
  • 控制空间：表征控制量空间的参数化。控制空间适用于动力学系统，对于集合层面的规划，无需考虑控制量问题
  • 采样器：用于从状态空间内产生不同的状态。对于基于控制的系统，使用独立的控制量采样机对控制空间进行采样
  • 状态有效性检测器：用于区分状态空间的有小部分和无效部分的算法。如：使用状态有效性检测器进行避障检测，以及速度和加速度超限检测等
  • 局部规划器：在集合层面进行规划时，局部规划用于进行状态点之间的插补

OMPL 中的规划算法都是在非常抽象的状态空间中工作。OMPL 中已实现了多种常用状态空间，例如：

  • SE(2)：平面平移及旋转
  • SE(3)：空间平移及旋转
  • R^n：ompl::base::RealVectorStateSpace
  • SO(2)：平面旋转
  • SO(3)：空间旋转
  • Time
  • Discrete：离散状态
  • Dubins：Dubins car’s 状态空间
  • ReedsShepp：Reeds-Shepp car’s 状态空间
  • OpenDE：Open Dynamics Engine
  • Constrained state spaces：约束状态空间

OMPL 中实现的控制空间包括：

  • R^n：ompl::control::RealVectorControlSpace
  • Discrete：ompl::control::DiscreteControlSpace
  • OpenDE：ompl::control::OpenDEControlSpace

OMPL 算法

基于采样运动规划：基本思想是使用图结构近似搜索空间的连续性，然后对搜索空间进行采样并选择样本点作为近似图的顶点，近似图内各顶点之间的连线表示有效路径。

OMPL 包含许多基于采样的算法的实现：

  • RPM
  • RRT
  • EST
  • SBL
  • KPIECE
  • SyCLOP

OMPL 规划算法分为两类：

  • 几何规划(Geometric planners)
  • 控制规划(Control-based planners)

几何规划

只考虑系统的几何和运动学约束。

  • 多任务规划(Multi-query planners)：为整个任务空间建立 roadmap

    • PRM(Probabilistic Roadmap Method)

    • LazyPRM

    • PRM*

    • LazyPRM

      • SPARS(SPArse Roadmap Spanner algorithm)
      • SPARS2

  • 单任务规划(Single-query planners)：通过可行的运动节点生成一棵树。他们的不同之处在于使用不同的启发式来生长树

    • RRT(Rapidly-exploring Random Trees)

      • RRT Connect(RRTConnect)
      • RRT*
      • Lower Bound Tree RRT(LBTRRT)
      • Sparse Stable RRT
      • T-RRT(Transition-based RRT)
      • Vector Field RRT
      • pRRT(Parallel RRT)
      • LazyRRT(Lazy RRT)

    • EST(Expansive Space Trees)

    • KPIECE(Kinematic Planning by Interior-Exterior Cell Exploration)

    • PDST(Path-Directed Subdivision Trees)

    • FMT*(Fast Marching Tree algorithm)

    • BFMT*(Bidirectional Fast Marching Tree)

  • 优化 planners(Optimizing planners)

    • PRM*
    • LazyPRM*
    • RRT*
    • RRT#
    • RRTX
    • Informed RRT*
    • Batch Informed Trees(BIT*)
    • Sparse Stable RRT
    • T-RRT
    • SPARS
    • SPARS2
    • FMT*
    • CForest
    • APS(AnytimePathShortening)

控制规划

这类 planners 依靠状态传播(state propagation)而不是简单的插值(interpolation)来产生运动。

  • RRT(Rapidly-exploring Random Trees)

  • SST(Sparse Stable RRT)

  • EST(Expansive Space Trees)

  • KPIECE(Kinodynamic Planning by Interior-Exterior Cell Exploration)

  • PDST(Path-Directed Subdivision Trees)

  • Syclop

    • Syclop using RRT as low-level planner
    • Syclop using EST as the low-level planner
