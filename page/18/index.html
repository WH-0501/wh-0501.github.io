catkin_make_isolated

当一个工作空间中包含不同环境的独立的项目时，可以使用该指令进行编译。为纯粹由 catkin 包构成的工作区提供调试，并允许用户构建包含 catkin 和 **非catkin **包的工作区。

非catkin包：编译脚本中不使用 Catkin cmake，而是简单的 CMake 或者其它的工具的程序包。

catkin build

主要区别在于 catkin build 获取的 isolated 环境。catkin build 使得整个 build configuration 更加独立，并且对配置文件的修改(添加/删除程序包，修改 cmake 变量等)更加健壮。

  • catkin build 并行编译相互独立的包，因此速度更快
  • catkin build 可在 workspace 的任何目录下执行，**catkin_make 只能在顶层目录下执行**
  • 编译指定包更加方便。catkin build ，或者，在要编译的包目录下执行 catkin build --this
  • catkin build 输出的日志更易读
  • 使用 catkin build，还可以获得许多其它有用的 catkin 子命令，例如:
    • catkin clean - 清除 build devel 及 install 目录
    • catkin list
    • catkin locate
    • catkin profile

命名规则

参考：ros Names

Graph Resource Names

Graph Resource Names 提供了一个分层的命名结构，该结构用于 ROS 计算图中的所有资源，如 nodes、参数、topics 和 services。

例如：

  • / : 全局命名空间
  • /foo
  • /stanford/robot/name
  • /wg/node

Graph Resource Names 是 ROS 中一个提供封装(encapsulation)的重要机制。每个资源定义在一个命名空间(namespace)中，而该命名空间有可能为多个资源所共享。通常，资源可在其 namespace 中创建资源，并且可以访问其 namespace 内或其 namespace 上一层的资源。可以建立属于不同 namespace 的资源之间的连接，这通常是通过在上层整合这两个 namespace 的代码来实现。

names 是相对解析的，因此资源不需要关注其所属的 namespace。

可用的 names

一个可用的名字有以下特征：

  • 以字母、’~’、’/‘ 开始
  • 后跟字母、数字、下划线或 ‘/‘

注意：

  • base names 不能使用 ‘/‘ 或 ‘~’

resolving

ROS 中有四种类型的 Graph Resource Names: base、relative、global、private

  • base
  • relative/name
  • /global/name
  • ~private/name

默认情况下，resolution is done relative to the node's namespace(resolution 是相对于节点的命名空间)。

没有指定任何 namespace 限定符的 names 属于 base names。**base names 实际上是 relative names 的一个子类，并且有着相同的 resolution 规则**。Base names 经常用于初始化节点名。

以 / 开头的 names 是全局的。应当尽量避免 Global names，因为其限制了代码的可移植性。

**以 ~ 开头的 names 是 private **。将节点的名字转换为 namespace。例如，/wg 命名空间中的节点 node1 的私有命名空间为 /wg/node1。Private names 用于通过参数服务器传递参数到一个指定节点。

示例:

Node Relative(default) Global Private
/node1 bar -> /bar /bar -> /bar ~bar -> /node1/bar
/wg/node2 bar -> /wg/bar /bar -> /bar ~bar -> /wg/node2/bar
/wg/node3 foo/bar -> /wg/foo/bar /foo/bar -> /foo/bar ~foo/bar -> /wg/node3/foo/bar

remapping

命令行启动节点时，可对 ROS 节点的名字进行重映射。

将 chatter 重映射为 /wg/chatter。

重映射匹配规则示例：

Node Namespace Remapping Argument Matching Names Final Resolved Name
/ foo:=bar foo,/foo /bar
/baz foo:=bar foo,/baz/foo /baz/bar
/ /foo:=bar foo,/foo /bar
/baz /foo:=bar /foo /baz/bar
/baz /foo:=/a/b/c/bar /foo /a/b/c/bar

Package Resource Names

该名字用于 ROS 文件系统级别的概念，用于简化指向磁盘文件和数据类型的过程。包资源名字非常简单: 在包名后增加资源名。例如，std_msgs/String 指 std_msgs 包中的 String 消息类型。

可以使用包资源名引用的一些ros相关文件包括:

  • Message types（msg）
  • Service types（srv）
  • Node types

可用 names：

  • 以字母（[a-z|A-Z]）开始
  • 后跟字母或数字、下划线、斜杠(/)
  • 最多有一个斜杠

全局名称

/ 表示全局名称。由斜杠分开的一系列命名空间，每个斜杠代表一级命名空间。基本名（base name）称描述资源本身。
例如：/teleop_turtle，该名称为全局名称，teleop_turtle 为基本名称。

相对名称

缺少全局名称的斜杠 /。

相对名称解析：将当前默认的命名空间的名称加在相对名称的前面，从而将相对名解析为全局名称。

设置默认命名空间：默认的命名空间是单独为每个节点设置的，而不是在系统范围进行。为节点选择一个不同的默认命名空间的最好的也是最常用的方法是在 launch 文件中使用命名空间属性(ns)。ROS 提供如下方式设置默认命名空间：

  • 只有当没有其他由 __ns 参数指定的默认命名空间时，ROS_NAMESPACE 环境变量才有效

私有名称

以 ~ 开始。和相对名称一样，私有名称并不能完全确定他们自身所在的命名空间，而是需要 ROS 客户端库将这个名称解析为一个全局名称。

与相对名称的主要差别在于：私有名称不是用当前默认命名空间，而是使用其节点名称作为命名空间。

匿名名称

一般用于节点名（匿名并不是指没有名字，而是指非用户指定而又没有语义信息的名字）。匿名名称的目的是使节点的命名更容易准守唯一性的规则。其思路是，当节点调用 ros::init 方法时可以请求一个自动分配的唯一名称。

注意：需要将 ros::init_options::AnonymousName 作为第四个参数传递给 ros::init 方法

AUTORCC

AUTORCC 是 boolean 值，指定 CMake 是否自动处理 Qt rcc 代码生成器，即：无需使用 QT4_ADD_RESOURCES()、QT5_ADD_RESOURCES()等命令。

若该属性为 ON，CMake 将处理在构建时添加为目标源的 .qrc 文件，并相应的调用 rcc。如果在创建目标时设置类该属性，则该属性由 CMAKE_AUTORCC 变量的值初始化。

默认情况下，AUTORCC 由自定义命令处理。 如果 .qrc 文件是 GENERATED，则使用自定义目标。

当存在多个同名的 .qrc 文件时，CMake 将为 rcc 生成未指定的唯一输出文件名。因此，如果需要使用 Q_INIT_RESOURCE() 或 Q_CLEANUP_RESOURCE()，则 .qrc 文件名必须是唯一的。

QT 接口

qt5_add_resources

使用 Resource Compiler（rcc）从 Qt 资源文件创建源代码。生成的源文件的路径已添加到中 。

qt5_add_big_resources

类似于 qt5_add_resources，但是直接生成目标文件（.o，.obj）而不是 C++ 源代码。这样可以嵌入更大的资源，而在这些资源中编译为 C++ 源代码然后再编译为二进制代码将非常耗时或占用大量内存。

注意：仅 CMake 3.9 及以上版本支持

qt5_wrap_cpp

创建用于在给定源文件上调用元对象编译器（moc）的规则。对于每个输入文件，将在构建目录中生成一个输出文件。生成文件的路径添加到中 <var>

为 cmake 设置快捷方式

基本语法

变量使用 ${} 方式取值，但是在 IF 控制语句中是直接使用变量名。

指令 ( 参数 1 参数 2 … ) 参数使用括弧括起，参数之间使用空格或分号分开。 以 ADD_EXECUTABLE 指令为例：ADD_EXECUTABLE(hello main.c func.c) 或者 ADD_EXECUTABLE(hello main.c;func.c) 。

指令是大小写无关的，参数和变量是大小写相关的。但，推荐全部使用大写指令。

cmake 语法灵活，例如： SET(SRC_LIST main.c) 也可写成 SET(SRC_LIST "main.c") ，但是当文件名中有空格时则必须使用双引号；还可以忽略源文件列表中的文件后缀， cmake 会自动在本目录查找。

当工程中有多个子目录时，每个子目录都需创建 CMakeLists.txt 文件

内部构建与外部构建

内部构建即在项目 CMakeLists.txt 同级目录进行配置及编译。外部构建即在项目外进行编译，例如：在项目中创建 <project>/build 目录，然后进入该目录执行 cmake .. && make 配置及配置。

外部构建时，<project>_SOURCE_DIRS 仍为项目根目录，而 <project>_BINRAY_DIR 则为 <project>/build

  • 根据参数规则，替换 @VAR@ 及 ${VAR} 变量

此外，会对 cmakedefine 进行替换

取决于是否在 CMake 中将 VAR 设置为任何未被 if() 命令视为 false 常量的值。即：cmakedefine 会根据变量是否为真来变换。

与 #cmakedefine VAR ... 形式不同，在 #cmakedefine01 VAR 形式中，VAR 本身将扩展为 VAR 0 或 VAR 1 而不是被赋值 … 因此：

#cmakedefine01 VAR ... 形式的输入行将扩展为 #cmakedefine01 VAR ... 0 或 *#cmakedefine01 VAR ... 0*，这可能会导致未定义的行为。

3.10 版中的新功能：结束行（#undef 注释除外）可以在 # 字符和 cmakedefine 或 cmakedefine01 单词之间使用空格或制表符进行缩进。 此空白缩进将保留在输出行中：

如果输入文件被修改，构建系统将重新运行 CMake 以重新配置文件并再次生成构建系统。生成的文件会被修改，并且只有在其内容发生更改时才会在后续 cmake 运行时更新其时间戳。

参数

  • input：input 文件路径。相对路径是根据 CMAKE_CURRENT_SOURCE_DIR 进行处理。 路径必须是文件，不能是目录
  • output：输出文件或路径。相对路径是根据 CMAKE_CURRENT_BINARY_DIR 进行处理
  • NO_SOURCE_PERMISSIONS: V 3.19 新增。不保留 文件的权限。默认为 644(即: -rw-r–r–)
  • USE_SOURCE_PERMISSIONS: V 3.20 新增。保留 文件的权限。当不指定三个权限相关参数时，默认执行改选项
  • FILE_PERMISSIONS : V 3.20 新增。忽略 文件的权限，并设置为 permissions 指定的权限
  • COPY: 仅拷贝 文件的内容到 文件，不进行变量替换
  • ESCAPE_QUOTES: 使用反斜杠来进行转义
  • @ONLY: 限制替换，仅替换 @VAR@ 变量，不替换 ${VAR} 变量
  • **NEWLINE_STYLE <style>**：指定 文件的新行格式，如：Unix 中使用 \n，windows 中使用 \r\n

注意: COPYONLY 和 NEWLINE_STYLE 冲突，不能同时使用。

Docker

发表于 2024-09-07 更新于 2024-09-22 分类于 Docker 本文字数： 3.7k 阅读时长 ≈ 7 分钟

概述

BUILD, SHIP, RUN

即：build – Docker 可以轻易构建用户的应用

SHIP – Docker 可以将应用快速分发

RUN – Docker 有能力秒级启动应用

基于GO开发，遵循Apache2.0。

Docker是一个基于LXC（即:linux container）技术之上构建的Container容器引擎。

Docker 使用客户端-服务器 (C/S) 架构模式，使用远程API来管理和创建Docker容器。

Docker特性

  • 速度飞快以及优雅的隔离框架
  • 物美价廉
  • CPU/内存的低消耗
  • 快速开/关机
  • 跨云计算基础构架

Docker组件与元素

Docker有三个组件和三个基本元素。

三个组件分别是：

  • Docker Client 是用户界面，它支持用户与Docker Daemon之间通信。
  • Docker Daemon 运行于主机上，处理服务请求。
  • Docker Index 是中央registry，支持拥有公有与私有访问权限的Docker容器镜像的备份。

三个基本要素分别是：

  • Docker Containers 负责应用程序的运行，包括操作系统、用户添加的文件以及元数据。
  • Docker Images 是一个只读模板，用来运行Docker容器。
  • DockerFile 是文件指令集，用来说明如何自动创建Docker镜像。

Docker的优势

  • Docker层是基于容器技术的轻量级虚拟化，没有Hypervisor层，省去了Hypervisor的开销
  • 对层级的创新应用，通过写入自己特有的内容后添加新的镜像层，新增的镜像层和下层的镜像一起又可以作为基础镜像被更上层的镜像使用。这种特性可以极大地提高磁盘利用率。同时，多个容器使用同一个基础镜像时，可极大地减少内存占用等方面，因为不同的容器访问同一个文件时，只会占用一份内存。

Docker 可视化工具

  • Portainer
  • LazyDocker

目标

实现轻量级的操作系统虚拟化解决方案。

Docker产生的目的主要是为了解决如下问题。

环境管理复杂

从各种OS到各种中间件再到各种App，一款产品能够成功发布，作为开发者需要关心的东西太多，且难于管理，这个问题在软件行业中普遍存在并需要直接面对。Docker可以简化部署多种应用实例的工作，比如Web应用、后台应用、数据库应用、大数据应用比如Hadoop集群、消息队列等等都可以打包成一个Image部署。

云计算时代的到来

AWS的成功, 引导开发者将应用转移到云上, 解决了硬件管理的问题，然而软件配置和管理相关的问题依然存在 (AWS cloudformation是这个方向的业界标准)。Docker的出现正好能帮助软件开发者开阔思路，尝试新的软件管理方法来解决这个问题。

虚拟化手段的变化

云时代采用标配硬件来降低成本，采用虚拟化手段来满足用户按需分配的资源需求以及保证可用性和隔离性。然而无论是KVM还是Xen，在 Docker 看来都在浪费资源，因为用户需要的是高效运行环境而非OS, GuestOS既浪费资源又难于管理, 更加轻量级的LXC更加灵活和快速。

与虚拟机的区别：

  • 虚拟机是用来进行硬件资源划分的完美解决方案，它利用了硬件虚拟化技术。通过一个 Hypervisor 层来实现对资源的彻底隔离；而容器则是操作系统级别的虚拟化，利用的是内核的 Cgroup 和 Namespace 特性，完全是通过软件来实现，仅仅是进程本身就可以与其它进程隔离开，不需要任何辅助
  • Docker 容器与主机共享操作系统内核，不同的容器之间可以共享部分系统资源，因此容器更加轻量级，消耗的资源也更少。而虚拟机会独占分配给自己的资源，几乎不存在资源共享，各个虚拟机实例之间近乎完全隔离，因此虚拟机更加重量级，也会消耗更多的资源
  • 容器被用来设计运行单进程。但 docker exec 命令已解决了该问题

LXC的便携性

LXC在 Linux 2.6 的 Kernel 里就已经存在了，但是其设计之初并非为云计算考虑的，缺少标准化的描述手段和容器的可便携性，决定其构建出的环境难于分发和标准化管理(相对于KVM之类image和snapshot的概念)。Docker就在这个问题上做出了实质性的创新方法。

容器首先必须是一个相对独立的运行环境，且在一个容器环境内，应该最小化其对外界的影响。
