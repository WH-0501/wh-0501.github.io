此外，参数 warehouse_host 和 warehouse_port 配置连接细节。对于 SQLite 插件，warehouse_host 包含数据库文件的路径，而 warehouse_port 未使用。

Subframes 是在 CollisionObjects 上定义的帧。可以用来定义放置在场景中的物体上的兴趣点，比如瓶子的开口、螺丝刀的尖端或螺钉的头部。它们可以用于计划和编写机器人指令，如“拿起瓶子，然后移动水龙头壶嘴下的开口”，或“拿起螺丝刀，然后把它放在螺钉头部上方”。

mesh filter

发表于 2024-09-17 更新于 2024-09-22 分类于 机械臂 ， MoveIt 本文字数： 184 阅读时长 ≈ 1 分钟

MoveIt 的网格过滤器功能从点云中移除机器人的几何形状。如果机器人的手臂在深度传感器的视图中，则与手臂相关的点将从点云中减去。

这是通过将原始点云、机器人的变换(\tf)和机器人的 URDF 作为输入来实现的。然后，过滤器发布一个修改后的点云，该点云不包含与当前机器人状态重叠的点。

要连接多个轨迹并一次规划轨迹，可以使用序列功能。这减少了规划开销，并允许遵循预先描述的路径而不停留在中间点。

请注意: 序列命令允许包含多个组的命令(例如“机械手”，“夹持器”)

User interface sequence capability

一个专门的 MoveIt 功能将 moveit_msgs::MotionSequenceRequest 作为输入。该请求包含如上所述的后续目标列表和附加的 blend_radius 参数。如果给定的以米为单位的 blend_radius 大于零，则将相应的轨迹与以下目标合并在一起，以使机器人不会停在当前目标处。当 TCP 比给定的 blend_radius 更接近目标时，允许它朝下一个目标前进。当离开当前目标周围的球体时，机器人会返回到它在没有混合的情况下所采用的轨迹。

详细实现参考：MotionBlendAlgorithmDescription.pdf

URDF

MoveIt 以 URDF(通用机器人描述格式)开始，这是 ROS 中描述机器人的原生格式。

介绍确保生成的 URDF 可以与 MoveIt 一起使用的技巧。

soft_lower_limit 字段和 soft_upper_limit 字段指定该关节的关节位置限制。MoveIt 将把这些限制与 URDF 中指定的关节的硬限制进行比较，并选择更保守的限制。

注意：如果将 soft_lower_limit 和 soft_upper_limit 设置为 0.0，可能导致关节无法运动。

碰撞检查

MoveIt 使用 URDF 中指定的 mesh 进行碰撞检查。URDF 允许指定两组 mesh 分别用于可视化和碰撞检查。一般来说，可视化 mesh 可以很详细，但是 collision mesh 应该不那么详细。Mesh 中三角形的数量会影响机器人连接碰撞检查所需的时间。整个机器人的三角形数量应该在几千的数量级。

SRDF 概述

Semantic Robot Description Format，描述机器人的语义信息。

这种格式的目的是表示 URDF 文件中没有的关于机器人的信息，但它对各种应用程序都很有用。其目的是包含具有语义方面的信息。

SRDF 补充了 URDF，并指定了：

  • joint groups
  • default robot configurations
  • additional collision checking information
  • additional transforms

推荐使用 MoveIt! Setup Assistant 生成 SRDF 文件。

Virtual Joints

URDF 中定义机器人上真实存在的关节信息，SRDF 中则可通过 添加虚拟关节。该功能常用语将机器人的 root joint 与 world 坐标系相关联。

Passive Joints

即：被动关节。指不受直接控制的关节。例如：差动机器人的从动轮。它们在 SRDF 中单独指定，以确保运动规划或 control pipelines 中的不同组件知道无法直接控制此关节。

Groups

MoveIt! 的核心概念，MoveIt! 始终作用于特定 group。MoveIt! 中所有运动都是基于特定的 group 进行规划，每次规划只会控制该规划组包含的关节，而保持其余的关节不变。Group 是关节和 link 的集合。可以通过以下方式定义 group：

  • Collection of joints：通过一组 joints 来定义一个 group。每个 joint 的所有 child link 都自动包含在组中

  • Collection of links：通过一组 links 来定义一个 group。所有 link 的 parent joint 都会被自动添加到 group 中

  • Serial Chain：通过串联 link 来定义一个 group，用 base link 和 tip link 来指定 serial chain

    • base link 是该 chain 中的第一个 joint 的 parent link
    • tip link 是该 chain 中最后一个 joint 的 child link

  • Collection of Sub-Groups：一个 group 也可以是多个其它 group 的集合。如：定义左臂规划组 left_arm、右臂规划组 right_arm，并在 arms 组中包含这两个 group，实现同时控制左右臂

End-Effectors

末端执行器是一种特殊的 group，区分与一般的 group。

定义末端执行器规划组时，要保证和与之关联的规划组没有共有的 link。

Self-Collisions

默认自碰撞矩阵生成器（Setup Assistant 的一部分）搜索机器人上可以安全地从碰撞检查中禁用的 links 对，从而减少运动规划处理时间。当这些links 对处于以下状态时将被禁用：

  • Always in collision：始终处于碰撞中
  • Never in collision：从不碰撞
  • in collision in the robot’s Default position
  • Adjacent links：相邻 links

MoveIt! Setup Assistant 中可以通过 Default Self-Collision Matrix Generator 来检测机器人的自碰撞。

采样密度指定了多少个随机的机器人位置来检查自碰撞。较高的密度需要更多的计算时间，而较低的密度有更大的可能性禁用不应该禁用的对。默认值是10000 个冲突检查。冲突检查是并行进行的，以减少处理时间。

Robot Poses

SRDF 中还可以存储机器人的固定位姿，并指定 id，方便程序中调用。例如：SRDF 的一个典型示例是定义机械手的 HOME 位置。

SRDF 语法

SRDF 文件中的标签

  • <**robot**>：

    • Purpose：SRDF 文档的 根 标签。所有信息均包含在该标签内

    • Elements：

      • <group>
      • <end_effector>
      • <virtual_joint>
      • <group_state>
      • <disable_collisions>

    • Attributes：

      • name：robot 名字，需与对应的 URDF 文件中同名

  • <group>

    • Purpose：一组 joints 和 links 的表示。这可以用于指定规划的自由度，定义手臂，末端执行器等

    • Elements：

      • <link>
      • <joint>
      • <chain>
      • <group>

    • Attributes：

      • name：group 名字

    • 注意：当一个组不包含任何其它 tags，只包含 name 属性时，被假定为具有该名称的组(在前面定义)的引用

  • <**group_state**>

    • Purpose：根据 joint value 为特定组定义命名状态。这对于定义像 “folded arms” 这样的状态很有用

    • Elements：<joint>

    • Attributes：

      • name：状态名
      • group：该状态对应的 group 名

    • 注意：为了使组状态有效，组中的所有 joint 都必须将其值指定为 group state 的一部分

  • <**link**>

    • Purpose：指定 中的 link

    • Elements：None

    • Attributes：

      • name：必须与 URDF 中的 link 名一致

    • 注意：当一个 link 包含在一个组中时，相应的父关节(如果存在)也是如此

  • <**joint**>

    • Purpose：指定 中的 joint

    • Elements：None

    • Attributes：

      • name：必须与 URDF 中的 joint 名一致
      • value：只出现在 中，浮点值。当一个 joint 包含多个自由度时，其 value 属性是一个由空格分隔的浮点值数组

    • 注意：当一个 joint 包含在一个组中时，相应的子 link 也应包含

  • <**chain**>

    • Purpose：表示机器人的 kinematic chain

    • Elements：None

    • Attributes：

      • base_link：该 chain 的 root link
      • tip_link：该 chain 的最后一个 link

    • 注意：根据 chain 中的 links，隐式的定义了 chain 中的 joints。与 chain 对应的 joints 是 chain 中 links 的父关节

  • <**end_effector**>

    • Purpose：描述末端执行器的信息

    • Elements：None

    • Attributes：

      • name：末端执行器的名字
      • group：包含组成末端执行器的 links(和 joints)的组名
      • parent_link：末端执行器连接到的 link 名
      • parent_group：是包含 parent_link 的关节组的名称。例如，如果有一个 arm group、一个 hand group 和一个包含手臂和手部关节的 arm_hand 组，那么手部末端执行器的 parent_group 将是 arm

    • 注意：根据 chain 中的 links，隐式的定义了 chain 中的 joints。与 chain 对应的 joints 是 chain 中 links 的父关节

  • <**virtual_joint**>

    • Purpose：添加非真实存在于机器人上的关节，定义机器人 link 与外部参照系(相对于机器人而言被认为是固定的)之间的虚关节。通常，会利用虚拟关节将机器人的根关节 root joint 和 world 坐标系关联起来

    • Elements：None

    • Attributes：

      • name：虚拟关节的名字

      • child_link：连接机器人与环境的 link 的名称

      • parent_frame：虚拟关节的父 link

      • type：虚拟关节的类型。可以是

        • fixed(0 DOF)
        • floating(all 6 DOF)
        • planar(3 DOF: x,y,yaw)

    • 注意：当 标签用于通过名称引用现有关节时，也可以使用已定义的虚拟关节的名称

​

  • <**disable_collisions**>

    • Purpose：默认情况下，假定机器人的任何 link 都可能与机器人中的任何其他 link 发生冲突。此标签禁用指定 link 对之间的冲突检查

    • Elements：None

    • Attributes：

      • link1：

      • link2：

      • reason：可选。这两个 link 之间的冲突应该被禁用的原因

        • Adjacent：相邻 link
        • Never：永远不会发送碰撞
        • Default：在默认位姿会产生碰撞
        • Always：总是碰撞

  • <**passive_joint**>

    • Purpose：被动关节。默认情况下，假定由 URDF 指定的所有 joints 都处于活动状态，并且它们的状态可用于规划。如果某些关节不是这种情况，则可以使用此标签来指定。即：不受直接控制的关节，例如移动底盘的从动轮

    • Elements：None

    • Attributes：

      • name：passive joint 的名字

  • <**link_sphere_approximation**>

    • Purpose：此标记指定一组保守地近似 link 的球体。球体在与 URDF 中的碰撞几何体相同的坐标系中指定。该标签是可选的。如果 link 没有出现任何标记，则将使用包含 link 的碰撞几何图形的单个球体。如果标签出现并且仅包含半径为 0 的球体，则在进行基于球体的碰撞检测时将不考虑该链接

    • Elements：sphere

    • Attributes：

      • link：近似为球体的 link

  • <**sphere**>

    • Purpose：定义一个球体

    • Elements：None

    • Attributes：

      • center：3个用空格分隔的浮点值，指定球体中心 x y z
      • radius：球体半径，浮点值

Time Parameterization

发表于 2024-09-17 更新于 2024-09-22 分类于 机械臂 ， MoveIt 本文字数： 2.3k 阅读时长 ≈ 4 分钟

MoveIt! 目前主要是一个 kinematic motion planning

framework - 规划关节或末端执行器的位置，而不是速度或加速度。然而，MoveIt! 会利用后处理来时间参数化运动轨迹的速度和加速度值。

Speed Control

From File

默认情况下，MoveIt! 将关节轨迹的速度和加速度设置为机器人的 URDF 或 joint_limits.yaml 中允许的默认值。

joint_limits.yaml 由 setup assistant 生成，最初是 URDF 中值的副本。如果需要特殊约束，用户可以将这些值修改为小于原始 URDF 值。特定的关节属性可以通过 max_position、min_position、max_velocity、max_acceleration 来改变。关节限制可以通过 has_velocity_limits、has_acceleration_limits 键来打开或关闭。

运行时

运动轨迹的速度也可以在运行时修改。要在每个运动规划的基础上改变速度，可以设置两个缩放因子，如 MotionPlanRequest.msg 中所述。在 MoveIt MotionPlanning RViz 插件中也可以使用 spinbox 来设置这两个因子

Time Parameterization Algorithms

MoveIt 支持多种运动学轨迹后处理算法，以添加时间戳和速度/加速度值。目前在 MoveIt 中有三个默认可用的选项:

  • Iterative Parabolic Time Parameterization：即 iptp，调用 AddTimeParameterization
  • Iterative Spline Parameterization：即 isp，调用 AddIterativeSplineParameterization
  • Time-optimal Trajectory Generation：即 totg，调用 AddTimeOptimalParameterization。注意，需 melodic 及以上版本

Iterative Parabolic Time Parameterization

IPTP - 迭代抛物线时间参数化算法默认在 motion planning pipeline 中作为 planning request adapter 使用。尽管 MoveIt 使用的迭代抛物线时间参数化算法多年来已经被数百个机器人使用，但它存在已知的 bug。

使用 五次样条曲线 插补。

  • moveit 默认使用规划器，可以实现速度和加速度平滑，但无法避免加速度的抖动，详细参考 Improve time parameterization。
  • 该规划器等间距插补，平滑速度和加速度，适合用在低速精确轨迹控制下。

Iterative Spline Parameterization

迭代样条参数化算法与 pr382 合并为解决这些问题的。虽然初步的实验很有希望，但在完全替代迭代抛物时间参数化算法之前，正在等待来自社区的更多反馈。

三次样条曲线。

原理参考 Improved IPTP by fitting a cubic spline

Time-optimal Trajectory Generation

在 pr#809 和 pr#1365 中引入的时间最优轨迹生成算法可以产生非常平滑和连续的速度 profiles 轨迹。该方法是在原始轨迹上拟合路径段，然后从优化后的路径上采样新的路径点。这与严格的时间参数化方法不同，因为生成的路点可能在一定的误差范围内偏离原始轨迹。因此，在使用此方法时，可能需要额外的碰撞检查。

算法原理参考 Time-Optimal Path Following (July 2012)

  • 该规划器输出轨迹为等时间间距（其余为等距）
  • 该规划器在小间距下规划的速度和加速度较其余两种优化算法更加合理与平滑，在较大间距下三种规划算法效果差距不大，良好的规划间距会因机械臂参数的不同而有所不同
  • 在过小间距下(0.001s)插补，会出现运动规划不恒定，规划速度不平滑的现象，经测试在0.005s（不同机械臂不一致，同时需要根据控制器输出频率等确定）的规划间距下可以有较为稳定和理想的规划效果。

通过机器人上的 controllers 来配置 MoveIt。假设机器人为手臂提供了 FollowJointTrajectory action service，(可选地)为夹手提供了GripperCommand 服务。如果机器人不提供这个功能，推荐使用 ROS control 框架，可以很容易地在硬件通信层添加这个功能。

YAML 配置

首先在 robot_moveit_config/config 中创建 controllers.yaml，指定机器人的 controller 配置。例如：为 panda_arm 配置一个 FollowJointTrajectory action controller，并为 hand 配置 GripperCommand

FollowJointTrajectory 控制器接口

包含以下参数：

  • name：该 controller 的名字
  • action_ns：该 action 的命名空间
  • type：该 action 的类型
  • default：默认控制器是 MoveIt 选择用于与特定关节集通信的主控制器
  • joints：该接口控制的关节

GripperCommand 控制器接口

包含以下参数：

  • name：该 controller 的名字
  • action_ns：该 action 的命名空间
  • type：该 action 的类型
  • default：默认控制器是 MoveIt 选择用于与特定关节集通信的主控制器
  • joints：该接口控制的关节
  • parallel：当设置这个值时，关节的大小应该是 2，命令将是两个关节的和

轨迹允许执行时间参数

可选。对于每个控制器，可以选择设置 allowed_execution_duration_scaling 和 allowed_goal_duration_margin 参数。是全局参数 trajectory_execution/allowed_execution_duration_scaling 和 trajectory_execution/allowed_goal_duration_margin 的特定于控制器的重写。与全局值相反，特定于控制器的值不能在运行时动态重新配置。这些参数用于计算允许的轨迹执行持续时间，通过缩放预期执行持续时间并随后添加余量。如果超过这个持续时间，轨迹将被取消。控制器相关参数的设置方法如下

Hand-Eye Calibration

发表于 2024-09-17 更新于 2024-09-22 分类于 机械臂 ， MoveIt 本文字数： 1.3k 阅读时长 ≈ 2 分钟

ROS noetic 版本添加。

安装使用

MoveIt Calibration 包提供了进行手眼相机校准的插件和图形界面。可以对刚性安装在机器人底座架上的相机(eye_to_hand)和安装在末端执行器上的相机(eye_in_hand)进行校准。

RViz -> 菜单栏 Panels -> Add New Panel -> moveit_calibration_gui -> HandEyeCalibration.

然后创建一个视觉校准目标。该目标具有独特的模式，在图像数据中很容易识别，并且通过提供目标尺寸的测量，可以估计出目标在摄像机坐标系中的姿态。在进行手眼标定时，不需要知道目标的精确位置——只要目标静止在机器人的 base frame 内，手眼标定就可以从 5 个或更多位姿序列中估计出来。

在 Target 标签中，输入 Target Params 创建目标。然后使用 “Save target” 按钮保存目标图像，并打印出图像。可以随意使用目标参数进行实验，以了解它们如何影响目标，但一定要记住打印的目标所使用的参数——需要输入相同的参数才能识别目标。

目标必须是平的，才能被摄像机可靠地定位。把它放在平坦的表面上就足够了，也可以安装在木板上。测量标记的宽度(其中一个黑色方块的外部尺寸)，以及标记之间的间隔距离。在 Target Params 部分的相应框中输入这些值，单位为米。同时，在 Image topic 和 CameraInfo主题 下拉菜单中选择合适的主题。

几何背景

HandEyeCalibration 插件的 Context 标签中，包含进行校准所需的几何信息。

  • Sensor configuration：

    • Eye-in-hand
    • Eye-on-hand

  • Sensor frame：camera optical frame(相机光学坐标系)

  • Object frame：标定目标的坐标系

  • End-effector frame：末端执行器坐标系

  • Robot base frame：机械臂基座坐标系

FOV 部分在 RViz 中控制相机的视场渲染。要查看 FOV，添加 MarkerArray 显示，并将其设置为 /rviz_visual_tools 主题

采集数据

需要采集几个 samples 以确保良好的校准。机器人运动学提供了机器人底座上的末端执行器的姿态,并根据上面所述,估计了摄像机坐标系中的标定目标。如果目标在机器人 base 坐标系的姿态是准确的,只有对摄像机目标变换的单一观察才能在末端执行器框架中恢复相机的姿势。直接的 camera-to-end-effector 转换相当于复合相机到目标到基线-到末端的执行器变换。然而,一个更好的选择是将这些信息与几个姿势结合起来,以消除在这个等式的基础框架中设定的目标,正如Kostas Daniilidis所描述的那样。

traceroute&tracepath

traceroute 和 tracepath 均用于 确定数据包从源主机到目标主机所采取的路径。

两者的区别：

  • 工作原理：Traceroute 在数据包中加入一个 TTL（Time to Live）值，每次经过一个路由器时此值减一。当 TTL 的值减为零时，路由器应该丢弃数据包，然后向源主机发送一条 ICMP echo 消息。通过检测这些回显消息，Traceroute 工具就可以确定从源主机到目标主机所经过的所有路由器；Tracepath 使用 MTU 探测技术，自动检测通过每个路由器的最大传输单位（MTU），以便有效地确定路径
  • 使用方式：Traceroute 使用命令 traceroute 或 tracert，默认使用 ICMP 数据包，并且 需要管理员权限；Tracepath 使用命令 tracepath，它默认使用 UDP 数据包，并且不需要管理员权限
  • 结果显示：Traceroute 显示 每个路由器的 IP 地址、域名、RTT（Round Trip Time）和 TTL 值；Tracepath显示 每个路由器的 IP 地址、域名、RTT（Round Trip Time）和 MTU 值
  • 功能：Traceroute 可以使用 -n 选项来阻止 DNS 解析，以加快操作速度；Tracepath 可以使用 -p 选项来指定要使用的端口号

显示 *，可能防火墙封掉了 ICMP 的返回信息；或者添加 -I 选项。其它解决方法：

  • 配置 ICMP 超时报文功能：ip ttl-expires enable
  • 关闭 tracert 报文共计防范功能：undo firewall defend tracert enable 谨慎使用

netstat

netstat的输出结果可以分为两个部分:

  • Active Internet connections 有源TCP连接，其中”Recv-Q”和”Send-Q”指接收队列和发送队列。这些数字一般都应该是0。如果不是则表示软件包正在队列中堆积。这种情况只能在非常少的情况见到。
  • Active UNIX domain sockets 有源Unix域套接口(和网络套接字一样，但是只能用于本机通信，性能可以提高一倍)。

列名解释：

  • Proto：显示连接使用的协议。
  • RefCnt：表示连接到本套接口上的进程号。
  • Types：显示套接口的类型。
  • State：显示套接口当前的状态。
  • Path：表示连接到套接口的其它进程使用的路径名。

常用选项：

  • -t : 指明显示TCP端口，t 是 TCP 的首字母
  • -u : 指明显示UDP端口，u 是 UDP 的首字母
  • -l : 仅显示监听套接字，l 是 listening 的首字母
  • -p : 显示进程标识符和程序名称，每一个套接字/端口都属于一个程序，p是program的首字母
  • -n : 不进行DNS轮询，显示IP(可以加速操作)，n是numeric的首字母，以数字形式显示地址和端口号。否则显示主机名等信息
  • -a : 显示所有连接和侦听端口，a是all的首字母
  • -o : 显示拥有的每个连接关联的进程id，o是own的首字母
  • -c : 持续输出
  • -x : Unix 套接字
