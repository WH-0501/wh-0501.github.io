Kernel Module Signing

https://docs.nvidia.com/jetson/archives/r36.3/DeveloperGuide/SD/Security/SecureBoot.html#kernel-module-signing

内核模块签名工具在 install 期间对模块进行签名，然后在加载模块时检查签名。这可以通过禁止加载未签名的模块或使用无效密钥签名的模块来提高内核安全性。

以下是内核模块签名的内核配置选项：

  • 要启用内核模块签名验证，需启用 CONFIG_MODULE_SIG

  • 要选择内核模块签名验证模式，请将 CONFIG_MODULE_SIG_FORCE 设置为以下选项之一：

    • off：宽容模式
      • 如果模块已签名，则必须具有受信任的签名
      • 如果模块未签名，则可以加载，并且内核被标记为受污染
    • on：限制模式
      • 模块只有在使用受信任的签名签名后才能加载
      • 其他模块将生成错误

  • 要在构建时启用自动内核模块签名，要设置 CONFIG_MODULE_SIG_ALL

默认情况下，即使启用了内核模块签名验证，内核模块在构建时也不会签名。

  • 要指定签名密钥，需使用自己的 PEM 格式私钥设置 CONFIG_MODULE_SIG_KEY
    • 默认情况下，如果 CONFIG_MODULE_SIG_KEY="certs/signing_key.pem" 未更改，则内核会自动生成用于内核模块签名的 PEM 格式签名密钥
    • 将 CONFIG_MODULE_SIG_KEY 设置为除 certs/signing_key.pem 默认值以外的值会禁用签名密钥的自动生成，并允许使用选择的密钥对内核模块进行签名。

CONFIG_SYSTEM_TRUSTED_KEYS 内核选项也可以设置为包含其他证书的 PEM 编码文件的文件名。它是编译到内核中的 X.509 证书，用于对内核构建时未签名的模块进行内核模块验证。

bumblebee

主要应用场景是 Intel + Nvidia 双显卡笔记本，默认关闭 N 卡，降低功耗，当需要更强图形渲染功能时开启 Nvidia 显卡。

安装 bumblebee 后，默认使用集显。当使用 optirun 或 primusrun 时，会使用独显。

bumblebee 安装的是 NVIDIA 闭源驱动。

bumblebee 安装时会默认安装 bbswitch，没有bbswitch（也没启用nouveau）的话，独显处于常开状态，不仅费电，而且起不到任何作用。bbswitch 的作用是控制 optimus 独显的电源。（注：optimus 是 NVIDIA 针对笔记本电脑的显示切换技术，可以实现独显与集显之间无缝、自动、实时的切换（使用者无需重启或手动调整），从而实现降低功耗）

同时，bumblebee 可以在用户层直接手动调用独显来运行程序，程序关闭后独显也自动关闭。

bumblebee vs Prime

prime 同时只能用一种显卡，要么独显要么集显；而 bumblebee 实现了两种显卡同时交互使用，互不影响。

参数说明：

  • REQUIRED 选项表示：如果包未找到，则 cmake 的过程会终止，并输出警告信息；
  • EXACT 选项表示版本号必须精确匹配；

module 与 config 模式

find_package 在搜索 package 时，有两种模式：”Module“ 模式和 “Config“ 模式

  • Module 模式：该模式下，cmake 搜索所有名为 Find.cmake 的文件（例如：FindProtobuf.cmake）。这些文件的路径由 CMAKE_MODULE_PATH 变量指定。若找到该文件，则读取并处理。该模式负责查找 package、检查版本并生成必要的信息。许多模块都只提供了有限的信息，而没有版本化相关的信息。若未找到任何模块，且未明确指定 MOUDLE 选项，则进入 Config 模式

  • Config 模式：完整的 Config 模式的命令格式为：

    CONFIG 选项可用于明确指定跳过 Module 模式，直接进入 Config 模式，NO_MODULE 选项与 CONFIG 选项类似。Config 模式尝试查找一个由待查找的包提供的配置文件的位置。包含该文件的路径会被存储在名为 _DIR 的缓存变量中。默认情况下，该命令搜索名为 的包。若指定了 NAMES 选项，则其后的 names 参数会替代 。该指令为每个指定的 name 参数搜索名为 Config.cmake 或 -config.cmake 的文件。可通过 CONFIGS 选项改变可能的配置文件的名字。如果找到了配置文件，它将会被 CMake 读取并处理。由于该文件是由包自身提供的，它已经知道包中内容的位置。配置文件的完整路径存储在 cmake 的 _CONFIG 变量中。所有 CMake 要处理的配置文件将会搜索对应包的安装信息，并且将该安装匹配的 appropriate version 存储在 cmake 变量 _CONSIDERED_CONFIGS 中，与之相关的版本号（associated version）将被存储在 _CONSIDERED_VERSIONS 中。

target_include_directories

指定编译参数给定的 target 时需要包含的目录。该 target 必须经过 add_executable 或 add_library 命令编译，不能是 IMPORTED target。

与 include_dirctories 区别

include_directories 影响目录范围。该 CMakeLists.txt 中的所有 targets，以及在其调用点之后添加的所有子目录中的目标，均会将 include_directories 指定的目录包含到其路径中

target_include_directories(target x/y) 则具有目标范围，将目录 x/y 添加到目标 target 的包含路径。

如果所有目标使用相关的包含目录，则需要用 include_directories，如果路径特定于某个目标，或想更好地控制路径的可见性，则需要 target_include_directories。

target_sources

指定编译给定 target 时用到的源文件。其中 target 必须由 add_executable 或 add_library 之类的命令创建，不能为 ALIASA 目标。

Graph Resource Names

Graph Resource Names 提供了一个分层的命名结构，该结构用于 ROS 计算图中的所有资源，如 nodes、参数、topics 和 services。

例如：

  • / : 全局命名空间
  • /foo
  • /stanford/robot/name
  • /wg/node

Graph Resource Names 是 ROS 中一个提供封装(encapsulation)的重要机制。每个资源定义在一个命名空间(namespace)中，而该命名空间有可能为多个资源所共享。通常，资源可在其 namespace 中创建资源，并且可以访问其 namespace 内或其 namespace 上一层的资源。可以建立属于不同 namespace 的资源之间的连接，这通常是通过在上层整合这两个 namespace 的代码来实现。

names 是相对解析的，因此资源不需要关注其所属的 namespace。

可用的 names

一个可用的名字有以下特征：

  • 以字母、’~’、’/‘ 开始
  • 后跟字母、数字、下划线或 ‘/‘

注意：

  • base names 不能使用 ‘/‘ 或 ‘~’

resolving

ROS 中有四种类型的 Graph Resource Names: base、relative、global、private

  • base
  • relative/name
  • /global/name
  • ~private/name

默认情况下，resolution is done relative to the node's namespace(resolution 是相对于节点的命名空间)。

没有指定任何 namespace 限定符的 names 属于 base names。**base names 实际上是 relative names 的一个子类，并且有着相同的 resolution 规则**。Base names 经常用于初始化节点名。

以 / 开头的 names 是全局的。应当尽量避免 Global names，因为其限制了代码的可移植性。

**以 ~ 开头的 names 是 private **。将节点的名字转换为 namespace。例如，/wg 命名空间中的节点 node1 的私有命名空间为 /wg/node1。Private names 用于通过参数服务器传递参数到一个指定节点。

示例:

Node Relative(default) Global Private
/node1 bar -> /bar /bar -> /bar ~bar -> /node1/bar
/wg/node2 bar -> /wg/bar /bar -> /bar ~bar -> /wg/node2/bar
/wg/node3 foo/bar -> /wg/foo/bar /foo/bar -> /foo/bar ~foo/bar -> /wg/node3/foo/bar

全局名称

/ 表示全局名称。由斜杠分开的一系列命名空间，每个斜杠代表一级命名空间。基本名（base name）称描述资源本身。

相对名称

私有名称

以 ~ 开始。和相对名称一样，私有名称并不能完全确定他们自身所在的命名空间，而是需要 ROS 客户端库将这个名称解析为一个全局名称。

与相对名称的主要差别在于：私有名称不是用当前默认命名空间，而是使用其节点名称作为命名空间。

AUTORCC

AUTORCC 是 boolean 值，指定 CMake 是否自动处理 Qt rcc 代码生成器，即：无需使用 QT4_ADD_RESOURCES()、QT5_ADD_RESOURCES()等命令。

若该属性为 ON，CMake 将处理在构建时添加为目标源的 .qrc 文件，并相应的调用 rcc。如果在创建目标时设置类该属性，则该属性由 CMAKE_AUTORCC 变量的值初始化。

默认情况下，AUTORCC 由自定义命令处理。 如果 .qrc 文件是 GENERATED，则使用自定义目标。

当存在多个同名的 .qrc 文件时，CMake 将为 rcc 生成未指定的唯一输出文件名。因此，如果需要使用 Q_INIT_RESOURCE() 或 Q_CLEANUP_RESOURCE()，则 .qrc 文件名必须是唯一的。

QT 接口

qt5_add_resources

使用 Resource Compiler（rcc）从 Qt 资源文件创建源代码。生成的源文件的路径已添加到中 。

qt5_add_big_resources

类似于 qt5_add_resources，但是直接生成目标文件（.o，.obj）而不是 C++ 源代码。这样可以嵌入更大的资源，而在这些资源中编译为 C++ 源代码然后再编译为二进制代码将非常耗时或占用大量内存。

注意：仅 CMake 3.9 及以上版本支持

qt5_wrap_cpp

创建用于在给定源文件上调用元对象编译器（moc）的规则。对于每个输入文件，将在构建目录中生成一个输出文件。生成文件的路径添加到中 <var>

为 cmake 设置快捷方式

基本语法

变量使用 ${} 方式取值，但是在 IF 控制语句中是直接使用变量名。

指令 ( 参数 1 参数 2 … ) 参数使用括弧括起，参数之间使用空格或分号分开。 以 ADD_EXECUTABLE 指令为例：ADD_EXECUTABLE(hello main.c func.c) 或者 ADD_EXECUTABLE(hello main.c;func.c) 。

指令是大小写无关的，参数和变量是大小写相关的。但，推荐全部使用大写指令。

cmake 语法灵活，例如： SET(SRC_LIST main.c) 也可写成 SET(SRC_LIST "main.c") ，但是当文件名中有空格时则必须使用双引号；还可以忽略源文件列表中的文件后缀， cmake 会自动在本目录查找。

当工程中有多个子目录时，每个子目录都需创建 CMakeLists.txt 文件

内部构建与外部构建

内部构建即在项目 CMakeLists.txt 同级目录进行配置及编译。外部构建即在项目外进行编译，例如：在项目中创建 <project>/build 目录，然后进入该目录执行 cmake .. && make 配置及配置。

外部构建时，<project>_SOURCE_DIRS 仍为项目根目录，而 <project>_BINRAY_DIR 则为 <project>/build

FILES 与 programs

两者指令使用方法相同，不同的是安装后的权限。

programs 权限为：WNER_EXECUTE, GROUP_EXECUTE, 和 WORLD_EXECUTE，即 755
