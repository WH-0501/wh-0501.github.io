target_include_directories

指定编译参数给定的 target 时需要包含的目录。该 target 必须经过 add_executable 或 add_library 命令编译，不能是 IMPORTED target。

与 include_dirctories 区别

include_directories 影响目录范围。该 CMakeLists.txt 中的所有 targets，以及在其调用点之后添加的所有子目录中的目标，均会将 include_directories 指定的目录包含到其路径中

target_include_directories(target x/y) 则具有目标范围，将目录 x/y 添加到目标 target 的包含路径。

如果所有目标使用相关的包含目录，则需要用 include_directories，如果路径特定于某个目标，或想更好地控制路径的可见性，则需要 target_include_directories。

target_sources

指定编译给定 target 时用到的源文件。其中 target 必须由 add_executable 或 add_library 之类的命令创建，不能为 ALIASA 目标。

ADD_CUSTOM_TARGET

增加一个没有输出的目标，使得它总是被构建

ADD_CUSTOM_COMMAND

向生成的构建系统添加自定义构建规则。

ADD_CUSTOM_COMMAND 有两个限制：

  • 只有在相同的 CMakeLists.txt 中，指定了所有依赖于其输出的目标时才有效
  • 对于不同的独立目标，使用 ADD_CUSTOM_COMMAND的输出可以重新执行定制命令。这可能会导致冲突，应该避免这种情况的发生

只有当构建的目标以 ADD_CUSTOM_COMMAND 生成的OUTPUT文件为源代码的情况下，ADD_CUSTOM_COMMAND 中指定的命令才会才会执行。

options：

  • APPEND：将 COMMAND 和 DEPENDS 选项值附加到指定的第一个输出的自定义命令。之前必须已经调用过具有相同输出的此命令
  • BYPRODUCTS：指定命令预期生成的文件。如果副产品(byproduct)名是相对路径，它将相对于当前源目录相对应的构建树目录进行解释。每个副产品文件都将自动标记为GENERATED源文件属性
  • COMMAND：指定要执行的命令。如果指定了多个 COMMAND，它们将按顺序执行，但不一定组成有状态的 shell 或 batch 脚本。可选的 ARGS 参数是为了向后兼容。
  • COMMENT：在构建时执行命令之前显示给定的消息
  • DEPENDS：指定命令所依赖的文件。每个参数都转换为依赖项。如果未指定 DEPENDS，则只要缺少 OUTPUT，该命令就会运行；如果该命令实际上并未创建 OUTPUT，则该规则将始终运行
  • COMMAND_EXPAND_LISTS：
  • IMPLICIT_DEPENDS：请求扫描输入文件的隐式依赖项。给定的语言指定应使用其相应依赖扫描器的编程语言(programming language).目前仅支持C和CXX语言扫描器。必须为IMPLICIT_DEPENDS列表中的每个文件指定语言。IMPLICIT_DEPENDS选项目前仅支持Makefile生成器，其它生成器将忽略该选项。此选项不能与DEPFILE选项同时指定。
  • JOB_POOL：为 Ninja 生成器指定一个 pool。不能与 USES_TERMINAL 一起使用(暗含 console pool)。
  • MAIN_DEPENDENCY：指定命令的主要输入源文件。每个源文件最多可以有一个命令将其指定为其主要依赖项。编译命令(即用于库或可执行文件)算作隐式主要依赖项，它会被自定义命令规范覆盖
  • OUTPUT：指定命令预期生成的输出文件。每个输出文件将自动标记为生成的源文件属性。如果自定义命令的输出实际上没有作为磁盘上的文件创建，则应该使用 SYMBOLIC 源文件属性对其进行标记。如果输出名称是相对路径，则其绝对路径将解析为相对于与当前源目录对应的构建树目录(CMAKE_CURRENT_BINARY_DIR)或当前源目录(CMAKE_CURRENT_SOURCE_DIR)。3.20 版本以后，输出参数可使用一组受限的生成器表达式
  • USES_TERMINAL：如果可能，该命令将被授予直接访问终端(terminal)的权限
  • VERBATIM：命令的所有参数都将为构建工具正确转义(escaped),以便调用的命令接收每个参数不变。建议使用VERBATIM，因为它可以实现正确的行为。当没有给出VERBATIM时，行为是特定于平台的
  • WORKING_DIRECTORY：使用给定的当前工作目录执行命令。如果是相对路径，会被解释为相对于当前源目录对应的构建树目录
  • DEPFILE：指定一个包含自定义命令依赖项的depfile。它通常由自定义命令本身发出。这个关键字只能在生成器支持的情况下使用

Build Events

将自定义命令添加到 target，如库或可执行文件。这对于在构建 target 之前或之后执行操作非常有用。该命令将成为 target 的一部分，并且仅在构建 target 本身时执行。如果 target 已构建，则不会执行该命令。

这将定义一个新命令，该命令将与构建指定的 <target> 关联。 必须在当前目录中定义；不能指定在其它目录中定义的 target。

命令何时执行由以下指定的选项决定:

  • PRE_BUILD：在 Visual Studio Generators 上，在 target 中执行任何其它规则之前运行。在其它生成器(generator)上,在 PRE_LINK 命令之前运行
  • PRE_LINK：在编译源代码之后但在链接二进制文件(binary)或运行静态库的库管理员(librarian)或存档器工具(archiver tool)之前运行。这不是为 add_custom_target 命令创建的 target 定义的
  • POST_BUILD：在执行 target 中的所有其它规则后运行

使用 TARGET 格式时，project 应始终指定上述三个关键字之一。出于向后兼容性(backward compatibility)的原因，如果没有给出这样的关键字，则假定为 POST_BUILD，但 project 应明确提供其中一个关键字以明确他们期望的行为。

AUTORCC

AUTORCC 是 boolean 值，指定 CMake 是否自动处理 Qt rcc 代码生成器，即：无需使用 QT4_ADD_RESOURCES()、QT5_ADD_RESOURCES()等命令。

若该属性为 ON，CMake 将处理在构建时添加为目标源的 .qrc 文件，并相应的调用 rcc。如果在创建目标时设置类该属性，则该属性由 CMAKE_AUTORCC 变量的值初始化。

默认情况下，AUTORCC 由自定义命令处理。 如果 .qrc 文件是 GENERATED，则使用自定义目标。

当存在多个同名的 .qrc 文件时，CMake 将为 rcc 生成未指定的唯一输出文件名。因此，如果需要使用 Q_INIT_RESOURCE() 或 Q_CLEANUP_RESOURCE()，则 .qrc 文件名必须是唯一的。

QT 接口

qt5_add_resources

使用 Resource Compiler（rcc）从 Qt 资源文件创建源代码。生成的源文件的路径已添加到中 。

qt5_add_big_resources

类似于 qt5_add_resources，但是直接生成目标文件（.o，.obj）而不是 C++ 源代码。这样可以嵌入更大的资源，而在这些资源中编译为 C++ 源代码然后再编译为二进制代码将非常耗时或占用大量内存。

注意：仅 CMake 3.9 及以上版本支持

创建用于在给定源文件上调用元对象编译器（moc）的规则。对于每个输入文件，将在构建目录中生成一个输出文件。生成文件的路径添加到中 <var>

为 cmake 设置快捷方式

基本语法

变量使用 ${} 方式取值，但是在 IF 控制语句中是直接使用变量名。

指令 ( 参数 1 参数 2 … ) 参数使用括弧括起，参数之间使用空格或分号分开。 以 ADD_EXECUTABLE 指令为例：ADD_EXECUTABLE(hello main.c func.c) 或者 ADD_EXECUTABLE(hello main.c;func.c) 。

指令是大小写无关的，参数和变量是大小写相关的。但，推荐全部使用大写指令。

cmake 语法灵活，例如： SET(SRC_LIST main.c) 也可写成 SET(SRC_LIST "main.c") ，但是当文件名中有空格时则必须使用双引号；还可以忽略源文件列表中的文件后缀， cmake 会自动在本目录查找。

当工程中有多个子目录时，每个子目录都需创建 CMakeLists.txt 文件

内部构建与外部构建

内部构建即在项目 CMakeLists.txt 同级目录进行配置及编译。外部构建即在项目外进行编译，例如：在项目中创建 <project>/build 目录，然后进入该目录执行 cmake .. && make 配置及配置。

外部构建时，<project>_SOURCE_DIRS 仍为项目根目录，而 <project>_BINRAY_DIR 则为 <project>/build

该属性的值存储在变量 <var> 中。如果没有找到目标属性，则行为取决于它是否被定义为 INHERITED 属性(参见 define_property())。非继承属性将 <var> 设置为 <var>-NOTFOUND，而继承属性将按照 define_property() 命令的描述搜索相关的父范围，如果仍然无法找到该属性，则将 <var> 设置为空字符串。

指令格式

configure_file 主要作用：

  • 将 <input> 文件内容复制到 <output>
  • 根据参数规则，替换 @VAR@ 及 ${VAR} 变量

此外，会对 cmakedefine 进行替换

取决于是否在 CMake 中将 VAR 设置为任何未被 if() 命令视为 false 常量的值。即：cmakedefine 会根据变量是否为真来变换。

与 #cmakedefine VAR ... 形式不同，在 #cmakedefine01 VAR 形式中，VAR 本身将扩展为 VAR 0 或 VAR 1 而不是被赋值 … 因此：

#cmakedefine01 VAR ... 形式的输入行将扩展为 #cmakedefine01 VAR ... 0 或 *#cmakedefine01 VAR ... 0*，这可能会导致未定义的行为。

3.10 版中的新功能：结束行（#undef 注释除外）可以在 # 字符和 cmakedefine 或 cmakedefine01 单词之间使用空格或制表符进行缩进。 此空白缩进将保留在输出行中：

如果输入文件被修改，构建系统将重新运行 CMake 以重新配置文件并再次生成构建系统。生成的文件会被修改，并且只有在其内容发生更改时才会在后续 cmake 运行时更新其时间戳。

参数

  • input：input 文件路径。相对路径是根据 CMAKE_CURRENT_SOURCE_DIR 进行处理。 路径必须是文件，不能是目录
  • output：输出文件或路径。相对路径是根据 CMAKE_CURRENT_BINARY_DIR 进行处理
  • NO_SOURCE_PERMISSIONS: V 3.19 新增。不保留 文件的权限。默认为 644(即: -rw-r–r–)
  • USE_SOURCE_PERMISSIONS: V 3.20 新增。保留 文件的权限。当不指定三个权限相关参数时，默认执行改选项
  • FILE_PERMISSIONS : V 3.20 新增。忽略 文件的权限，并设置为 permissions 指定的权限
  • COPY: 仅拷贝 文件的内容到 文件，不进行变量替换
  • ESCAPE_QUOTES: 使用反斜杠来进行转义
  • @ONLY: 限制替换，仅替换 @VAR@ 变量，不替换 ${VAR} 变量
  • **NEWLINE_STYLE <style>**：指定 文件的新行格式，如：Unix 中使用 \n，windows 中使用 \r\n

注意: COPYONLY 和 NEWLINE_STYLE 冲突，不能同时使用。
