命名规则

参考：ros Names

Graph Resource Names

Graph Resource Names 提供了一个分层的命名结构，该结构用于 ROS 计算图中的所有资源，如 nodes、参数、topics 和 services。

例如：

  • / : 全局命名空间
  • /foo
  • /stanford/robot/name
  • /wg/node

Graph Resource Names 是 ROS 中一个提供封装(encapsulation)的重要机制。每个资源定义在一个命名空间(namespace)中，而该命名空间有可能为多个资源所共享。通常，资源可在其 namespace 中创建资源，并且可以访问其 namespace 内或其 namespace 上一层的资源。可以建立属于不同 namespace 的资源之间的连接，这通常是通过在上层整合这两个 namespace 的代码来实现。

names 是相对解析的，因此资源不需要关注其所属的 namespace。

可用的 names

一个可用的名字有以下特征：

  • 以字母、’~’、’/‘ 开始
  • 后跟字母、数字、下划线或 ‘/‘

注意：

  • base names 不能使用 ‘/‘ 或 ‘~’

resolving

ROS 中有四种类型的 Graph Resource Names: base、relative、global、private

  • base
  • relative/name
  • /global/name
  • ~private/name

默认情况下，resolution is done relative to the node's namespace(resolution 是相对于节点的命名空间)。

没有指定任何 namespace 限定符的 names 属于 base names。**base names 实际上是 relative names 的一个子类，并且有着相同的 resolution 规则**。Base names 经常用于初始化节点名。

以 / 开头的 names 是全局的。应当尽量避免 Global names，因为其限制了代码的可移植性。

**以 ~ 开头的 names 是 private **。将节点的名字转换为 namespace。例如，/wg 命名空间中的节点 node1 的私有命名空间为 /wg/node1。Private names 用于通过参数服务器传递参数到一个指定节点。

示例:

Node Relative(default) Global Private
/node1 bar -> /bar /bar -> /bar ~bar -> /node1/bar
/wg/node2 bar -> /wg/bar /bar -> /bar ~bar -> /wg/node2/bar
/wg/node3 foo/bar -> /wg/foo/bar /foo/bar -> /foo/bar ~foo/bar -> /wg/node3/foo/bar

remapping

命令行启动节点时，可对 ROS 节点的名字进行重映射。

将 chatter 重映射为 /wg/chatter。

重映射匹配规则示例：

Node Namespace Remapping Argument Matching Names Final Resolved Name
/ foo:=bar foo,/foo /bar
/baz foo:=bar foo,/baz/foo /baz/bar
/ /foo:=bar foo,/foo /bar
/baz /foo:=bar /foo /baz/bar
/baz /foo:=/a/b/c/bar /foo /a/b/c/bar

Package Resource Names

该名字用于 ROS 文件系统级别的概念，用于简化指向磁盘文件和数据类型的过程。包资源名字非常简单: 在包名后增加资源名。例如，std_msgs/String 指 std_msgs 包中的 String 消息类型。

可以使用包资源名引用的一些ros相关文件包括:

  • Message types（msg）
  • Service types（srv）
  • Node types

可用 names：

  • 以字母（[a-z|A-Z]）开始
  • 后跟字母或数字、下划线、斜杠(/)
  • 最多有一个斜杠

roscpp API

参考：ros::names Namespace Reference

全局名称

/ 表示全局名称。由斜杠分开的一系列命名空间，每个斜杠代表一级命名空间。基本名（base name）称描述资源本身。
例如：/teleop_turtle，该名称为全局名称，teleop_turtle 为基本名称。

相对名称

缺少全局名称的斜杠 /。

相对名称解析：将当前默认的命名空间的名称加在相对名称的前面，从而将相对名解析为全局名称。

设置默认命名空间：默认的命名空间是单独为每个节点设置的，而不是在系统范围进行。为节点选择一个不同的默认命名空间的最好的也是最常用的方法是在 launch 文件中使用命名空间属性(ns)。ROS 提供如下方式设置默认命名空间：

  • 只有当没有其他由 __ns 参数指定的默认命名空间时，ROS_NAMESPACE 环境变量才有效

私有名称

以 ~ 开始。和相对名称一样，私有名称并不能完全确定他们自身所在的命名空间，而是需要 ROS 客户端库将这个名称解析为一个全局名称。

与相对名称的主要差别在于：私有名称不是用当前默认命名空间，而是使用其节点名称作为命名空间。

匿名名称

一般用于节点名（匿名并不是指没有名字，而是指非用户指定而又没有语义信息的名字）。匿名名称的目的是使节点的命名更容易准守唯一性的规则。其思路是，当节点调用 ros::init 方法时可以请求一个自动分配的唯一名称。

注意：需要将 ros::init_options::AnonymousName 作为第四个参数传递给 ros::init 方法 

示例

使用 rosrun 指令直接运行节点时，若不使用 _name 则默认节点名为 ros::init 中指定的名字；若使用 launch 文件启动节点，则节点名为 <node> 元素中 name 属性指定的值；若 name 属性值为空，则为匿名名称。
