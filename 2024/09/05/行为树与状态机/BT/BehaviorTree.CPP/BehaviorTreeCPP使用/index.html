BehaviorTreeCPP使用

发表于 2024-09-05 更新于 2024-09-22 分类于 行为树 阅读次数： 本文字数： 19k 阅读时长 ≈ 35 分钟

参考：

  • Behavior Trees in Robotics and AI
  • ROS-Behavior-Tree

BehaviorTree.CPP is now a more sophisticated version of the library Behavior-Tree（ROS-Behavior-Tree）。

提供创建 BT 的框架的 C++ 库。与其它的 BT 实现由如下优势：

  • 支持异步 actions，例如: non-blocking、a first-class citizen
  • 可以构建同时执行多个 actions 的 reactive behaviors
  • 允许使用 XML 在运行时创建 BT
  • 可以静态连接 TreeNodes，或将其转换为运行时加载的插件
  • 包括 logging/profiling 基础结构，允许用户可视化、记录、回放和分析状态转换

ROS2 navigation2 基于该库实现。

编译

支持 catkin(ros)、colcon(ros2)、conan、cmake 编译

Groot

创建自定义节点

继承 TreeNode

要创建自定义节点，默认方式（推荐方法）应继承 TreeNode 类。

实际通常是继承 TreeNode 的以下子类：

  • ActionNodeBase
  • ConditionNode
  • DecoratorNode
  • SyncActionNode

注意：

  • 所有 TreeNode 实例都有 name，其名字不要求唯一
  • 必须重载子类虚函数 tick()，且返回值必须为 SUCCESS/FAILURE/RUNNING
  • halt() 用于暂停正在执行的 action。同步 action 可无需该方法
  • 若有输入/输出，还需定义静态方法 providedPorts

建议在 tick() 函数中调用 getInput() 方法，而不是在类的构造函数中。

使用依赖注入

除了继承 TreeNode 之外，也可以使用依赖注入方式(dependency injection)创建 TreeNode 并指定函数指针。

使用依赖注入将 类成员函数 注册为异步行为

Dataflow,Ports及Blackboard

  • blackboard：是一种 key/value 存储结构，用于在所有节点间共享数据
  • Ports：是节点之间用来交换信息的一种机制
  • Ports 使用 blackboard 的相同 key 连接
  • 一个节点的端口数量、名称和类型必须在编译时知道(c++);端口之间的连接是在部署时完成的(XML)
  • value 可以是任何 C++ type(使用类似 std::any 的类型确认机制)

  • Input Port 可以为

    • 静态类型(如字符串)
    • 一个 pointer, 指向 blackboard 上的一个条目, 由一个 key 标识
    • 对于输入端口，可通过 getInput(key) 读取输入值
    • 输入类型 InputPort

  • Output ports

    • 输出类型为 OutputPort
    • 对于输出端口，可通过 setOutput 设置输出值
    • 此外，调试时，可以使用内置的动作 Script 将静态值写入条目

Script 是 BT.CPP 3.X 版本中 SetBlackboard 的替代。SetBlackboard 已弃用

blackboard

Blackboard 是一个全局共享的数据存储空间。Blackboard 通常用于存储任务执行过程中的状态信息、目标、条件等数据, 供 Behavior Tree 节点进行读取和修改。

在 Behavior Tree 中, Blackboard 可以用来实现任务之间的通信和共享数据。例如, 一个行为节点可以将某个条件的结果写入 Blackboard, 然后另一个行为节点可以读取这个条件的结果进行判断。

Blackboard 通常是一个键值对的数据结构, 可以存储各种类型的数据, 如布尔值、整数、浮点数、字符串等等。Behavior Tree 节点可以通过读取和修改 Blackboard 中的数据来实现任务的执行和决策。

Blackboard 提供数据集中式设计。

集中式设计会定义一个数据结构专门用于行为树的输入, 在行为树决策之前, 将需要的数据先从各模块中收集到该数据结构中, 然后传递给行为树。这种方式减少了输入和行为树之间的接口数量(只与预定义的数据结构通信), 但存在数据冗余的缺点。不过, 集中式的数据输入使得行为树的表现更像一个黑盒了。

该 Blackboard 对于行为树而言是只读的, 行为树不允许修改和添加任何信息到这块黑板上面。

但 BT 在使用过程中, 有时候确实需要在节点/行为树之间传递数据, 例如 sequence 节点, 其子节点之间通常会涉及到节点通信问题。此时会定义另一个 blackboard, 负责行为树/节点间的通信

这块 blackboard 是又可以读又可以写的, 为了防止黑板混乱的问题, 必须在使用时规定一些限制, 可以称之为黑板数据的“作用域”：

  • 全局域（G）：此数据可以给其他行为树访问
  • 行为树域（T）：此数据可以给行为树内的任意节点访问
  • 指定节点域（N）：此数据可以给指定的行为树内的某节点（可以是多个）访问

Blackboard 的使用可以让 Behavior Tree 更加灵活和智能化, 使得任务的执行和决策能够根据实时的情境和条件进行动态调整。

全局 blackboard

在 BehaviorTree.CPP 4.6.0 版本引入。BT.CPP 使用 “scoped blackboards” 以隔离每个子树。

尽管如此，在某些情况下，有一个真正的 global blackboard 是可取的，它可以从每个子树直接访问，而不需要重新映射。

其用途：

  • 不能共享的单例对象和全局对象
  • 机器人的全局状态
  • 在行为树之外写入/读取的数据，即在执行 tick 的主循环中

每个子树有自己的 blackboard。Blackboard 的父子关系与行为树的父子关系相同。

私有 blackboard 的生命周期受其相关联的子树影响。

ports vs blackboard

BT.CPP 提出了 Input/Output Ports，以替代 Blackboards。

更具体地说，Ports 是一种接口，它为 blackboard 添加了一层间接和额外的语义。

要理解为什么推荐使用 Ports 而不鼓励直接使用 Blackboard，应该首先了解 BehaviorTree.CPP 的一些核心原则。

节点组合及子树范围

理想情况下，我们希望提供一个平台，允许行为设计师构建由不同供应商/用户实现的树(即“组合节点”)。

但是，当直接使用 Blackboard 时，名称冲突将成为一个问题。

例如，节点 GraspObject 和 MoveBase 可能是由不同的人开发的，它们都是从 blackboard 取 target。不幸的是，它们有不同的含义，类型也不同：前者期望一个 3D 位姿，而后者是一个 2D 位姿。

Ports 提供了一个间接层次，也称为 remapping。意味着，无论在定义 Port 时使用哪个名称，都可以将其 remapping 到 XML 中的不同 blackboard 条目，而无需修改源代码。

总结:不要直接使用 Blackboard

Pre and Post conditions

BT.CPP 4.x 引入。可以在Node的实际 tick() 之前或之后运行的脚本。

脚本的目标不是编写复杂的代码，而只是提高树的可读性，并在非常简单的用例中减少对自定义c++节点的需求。

如果脚本太长，可能需要重新考虑使用它们的决定。

Pre conditions

  • _skipIf: 若条件成立，则跳过该节点
  • _failureIf: 若条件成立，则跳过并返回 FAILURE
  • _successIf: 若条件成立，则跳过并返回 SUCCESS
  • _while: 与 _skipIf 类似，但当条件变为 false 时，可能中断 RUNNING 节点

与 ROS2 集成

提供了一组现成的封装 BehaviorTree.ROS2，可用于快速实现与 ROS2 交互的 treenode。

在系统架构方面，应该记住:

  • 应该有一个集中的 ROS 节点 Coordinator，负责执行行为。这将进一步被称为 Task Planner，它将与 BT.CPP 一起实现
  • 系统的所有其他元素都应该是面向服务(service-oriented)的组件，并且应该将任何业务逻辑和决策委托给 Task Planner

异步BT::Action使用 rclcpp_action

与 ROS 交互的推荐方式是使用 rclcpp_action。

他们完美匹配，因为:

  • 它们的 API 是异步的，也就是说用户不需要担心创建一个单独的线程
  • 它们可以被中止，这是实现 TreeNode::halt() 和构建响应性行为所需要的功能
